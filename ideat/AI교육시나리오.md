## SECTION 02 — 컨텍스트 윈도우의 정체 `3분` 🖼️
> AI의 작업 메모리 개념과 구성 요소, 모델별 크기 비교

### AI에게는 기억이 두 종류 있습니다

| | 학습 데이터 | 컨텍스트 윈도우 |
|---|---|---|
| **성격** | 장기 기억 | 작업 메모리 |

### 컨텍스트 윈도우 안에 들어가는 것들

**항상 들어가는 것**
- ⚙️ **시스템 프롬프트** — 도구가 AI에게 주는 초기 지시
- 💬 **내 질문 (전체 누적)** — 이번 대화의 모든 입력
- 🤖 **AI 답변 (전체 누적)** — AI가 생성한 모든 응답

**추가될 수 있는 것**
- 📎 **첨부 파일** — 올린 문서, PDF, 엑셀
- 🖼️ **캡처 이미지** — 붙여넣은 스크린샷
- 🔧 **MCP 툴 결과** — 외부 도구 호출 및 응답값

> ⚠️ **핵심:** 모든 항목이 동일한 공간을 공유합니다.
> 파일이 크면 대화 가능한 공간이 줄어듭니다.

## SECTION 03 — 프롬프트 → 컨텍스트 엔지니어링으로 `3분`
> 질문법이 아닌 맥락 관리가 핵심입니다

### 패러다임 전환

```
과거: 프롬프트 엔지니어링  →  어떻게 질문할까?
지금: 컨텍스트 엔지니어링  →  무엇을 어떻게 채워넣을까?
다음: 하네스 엔지니어링    →  어떤 환경에서 일하게 할까?
```

| 단계 | 핵심 질문 | 비유 |
|---|---|---|
| 프롬프트 | 어떻게 말할까? | 질문을 잘 던지기 |
| 컨텍스트 | 무엇을 알려줄까? | 참고자료를 전달하기 |
| **하네스** | **어떤 환경을 만들까?** | **업무 공간 자체를 세팅하기** |

> 2026년 업계의 결론은 명확합니다:
> **어떤 AI 모델을 쓰느냐보다, 그 AI가 일하는 환경을 어떻게 만드느냐가 결과를 결정합니다.**
> 오늘 교육은 이 세 단계 중 **컨텍스트**와 **하네스**를 다룹니다.

---

### 전달 방식이 결과를 바꾼다

같은 AI에게 같은 일을 시켜도, **무엇을 함께 전달하느냐**에 따라 결과가 달라집니다.

| | 프롬프트만 | 컨텍스트 포함 |
|---|---|---|
| **입력** | "환불 정책 만들어줘" | "월 9,900원 구독형인데, 가입-환불 반복 악용이 30%야. 지난번에 7일 이내 전액 환불로 바꿨더니 1일 사용 후 환불 패턴이 늘었어. 그래서 결제 직후 24시간 이내만 허용하는 방향으로 판단했어." |
| **결과** | 어디서든 쓸 수 있는 일반 템플릿 | **우리 상황에 맞는** 정책안 |

단순히 현황 정보가 아닙니다.
이전에 **어떤 시도를 했고**, **어떤 결과가 있었고**, **그래서 어떤 판단을 내렸는지** — 이 맥락이 차이를 만듭니다.

> **AI는 받은 맥락을 증폭하는 도구입니다.**
> 좋은 맥락을 넣으면 좋은 결과가 나오고,
> 잘못된 맥락을 넣으면 틀린 결과를 그럴듯하게 포장합니다.

---

### 💡 컨텍스트가 충분한지 확인하는 법

> AI에게 줄 모든 정보를 인쇄해서 **옆 동료에게 건네보세요.**
> 그 동료가 요청한 일을 못 하면, AI도 못 합니다.
> AI는 마음을 읽지 못합니다. 당신에게 당연한 것을 **명시적으로** 전달해야 합니다.

---

### 컨텍스트가 가득 찰 때 3가지 선택

| 방법 | 행동 | 결과 |
|---|---|---|
| **Clear** | 새 대화창 열기 | 깨끗하게 리셋. 하지만 이전 맥락 전부 소멸 |
| **Compact** | Claude Code `/compact` | 대화를 요약 압축. 핵심 맥락은 유지 |
| **새 에이전트** | Cursor 새 Agent 창 | 깨끗하지만 역시 이전 기억 없음 |

> ❗ **핵심 문제:** 어떤 방식이든 새로 시작한 AI는 기억이 없습니다.
> 바이브코딩 세션에서 2시간 쌓아온 결정, 실패한 시도, 설계 이유가 전부 날아갑니다.

## SECTION 04 — 하네스(Harness): AI의 업무 환경을 설계하기 `3.5분` 🖼️
> 기억은 압축되고, 채팅창은 휘발됩니다. 환경을 만들어야 합니다

### 기억만으로는 부족한 이유

"머릿속에 있는 걸 AI한테 말로 전달하면 되지 않나?"라고 생각할 수 있습니다.

하지만 사람의 기억에는 구조적 한계가 있습니다.
경험을 저장할 때 **자세한 맥락은 버리고 결론만 압축**해서 기억합니다.

| 기억나는 것 | 기억 안 나는 것 |
|---|---|
| "그때 A로 결정했다" | 어떤 대안을 검토했는지 |
| | 왜 B가 아니라 A였는지 |
| | 그 판단의 근거가 뭐였는지 |

기억이 안 나면 AI에게 전달할 수 없고,
전달할 수 없으면 **교과서 답만** 돌아옵니다.

> 🗣 **강사 노트**
> "그때 왜 그렇게 결정했어요?" — 기억 안 남
> "비슷한 거 전에 했었는데..." — 기억 안 남
> "퇴사한 동료 업무 인수" — 지식 전부 증발
> 청중의 고개가 끄덕여지면 다음으로 넘어갑니다.

그래서 지식이 머릿속이 아닌 **별도 공간에 기록**되어 있어야 합니다.

---

### 하네스란 무엇인가

하네스(Harness)는 원래 말의 마구(馬具), 안전벨트를 뜻합니다.
AI에서 하네스란 **AI가 올바른 방향으로 일할 수 있도록 둘러싼 업무 환경 전체**를 말합니다.

2026년 초, OpenAI 팀이 5개월 동안 단 한 줄의 코드도 직접 쓰지 않고
100만 줄 이상의 소프트웨어를 AI로 만든 경험을 공개했습니다.
비결은 더 좋은 AI 모델이 아니라 **AI가 일하는 환경(하네스)을 철저히 설계한 것**이었습니다.

AI를 **새로 입사한 팀원**이라고 생각해보세요:

| 상황 | 하네스 없이 | 하네스 있으면 |
|---|---|---|
| 첫 출근 | "알아서 해봐" | 업무 매뉴얼 + 온보딩 문서 제공 |
| 업무 중 | 실수해도 아무도 안 알려줌 | 체크리스트가 자동으로 검토 |
| 시간이 지나면 | 문서와 현실이 안 맞아짐 | 주기적으로 정리·갱신 |

위 표의 오른쪽 열, 그 세 가지가 바로 하네스의 구성 요소입니다.

---

### 하네스의 세 가지 요소

#### ① 업무 매뉴얼 — 맥락을 기록으로 남기기

대화는 휘발되지만 파일은 남습니다.
새 AI가 시작될 때 이 파일을 읽으면 즉시 맥락을 파악합니다.

| 파일 | 역할 |
|---|---|
| `CLAUDE.md` / `Cursor Rules` | AI가 매 세션 자동으로 읽는 기본 규칙과 배경지식 |
| **`SKILL.md`** | 특정 업무를 위한 맥락 + 지시 패키지 → `@파일`로 주입 |
| `_context/` 폴더 | 회의록, 결정 이력, 요구사항 등 참고 자료 모음 |

> "스킬 파일을 쓴다는 것
> = 파일 안의 텍스트를 새 에이전트의 컨텍스트 윈도우에 **주입**하는 것
> = AI에게 매번 온보딩 시키는 대신, 업무 매뉴얼을 건네는 것"

단순한 현황 정보가 아닙니다.
이전에 **어떤 시도를 했고**, **어떤 결과가 있었고**, **그래서 어떤 판단을 내렸는지** — 이 판단의 이력이 핵심입니다.

---

#### ② 가드레일 — "이것만은 반드시" 규칙 세우기

AI가 아무리 똑똑해도, 우리 팀만의 규칙을 모르면 엉뚱한 방향으로 갑니다.
**명확한 규칙을 세워두면 AI가 스스로 지키면서 일합니다.**

| 업무 상황 | 규칙 파일에 적을 내용 |
|---|---|
| 기획서 작성 | "목적 → 기능 정의 → WBS → 리스크 순서로 작성할 것" |
| 고객 응대 | "환불은 24시간 이내만 가능. 이 정책은 절대 변경 불가" |
| 보고서 작성 | "수치에는 반드시 출처 명시. 추정치는 '추정' 표기" |
| 콘텐츠 작성 | "우리 브랜드 톤은 ~입니다체. 이모지 사용 금지" |

> 💡 **역설적이지만 울타리가 있어야 그 안에서 마음껏 뛰어놀 수 있습니다.**
> AI도 마찬가지입니다. 규칙이 명확할수록 AI는 그 안에서 더 자유롭고 정확하게 일합니다.
> 이것이 하네스 엔지니어링의 핵심 원리입니다.

---

#### ③ 정기 점검 — 쌓인 것 정리하기

AI와 오래 일하다 보면 문서와 현실이 어긋나기 시작합니다.
깨끗한 책상에서 일이 잘 되듯, **정리된 환경에서 AI도 잘 일합니다.**

| 증상 | 처방 |
|---|---|
| 바뀐 결정이 문서에 반영 안 됨 | 주 1회 `_context/` 폴더 업데이트 |
| 안 쓰는 규칙이 계속 적용됨 | 규칙 파일에서 더 이상 필요 없는 항목 삭제 |
| 스킬 파일이 현재 업무와 안 맞음 | 스킬 파일 갱신 또는 새로 작성 |

> 🗣 **강사 노트**
> "정리 안 된 책상에서 일하면 효율이 떨어지잖아요? AI도 같습니다.
> 쌓아둔 문서가 오래되면 오히려 AI를 혼란스럽게 만듭니다.
> 주 1회 5분만 투자해서 정리하세요."

## SECTION 05 — Cursor 실무 팁 `2분`
> 비개발자도 바로 쓸 수 있는 컨텍스트 관리 루틴

### 컨텍스트 채우는 방법

| 명령 | 설명 |
|---|---|
| `@Cursor Rules` | 매 대화에 자동 주입되는 상시 규칙. 프로젝트 규칙, 내 업무 방식을 여기에 정의 |
| `@Past Chats` | 이전 대화 참조. 맥락을 이어가고 싶을 때 과거 결정사항 불러오기 |
| `@파일` / `@폴더` | 내가 원하는 것만 골라서 주입. 불필요한 것은 넣지 않아야 컨텍스트 절약 |

---

### 언제 새 에이전트를 열어야 하나

| 신호 | 설명 |
|---|---|
| 신호 1 | Cursor 하단 컨텍스트 바 토큰 사용량이 **절반을 넘었을 때** |
| 신호 2 | **주제가 바뀔 때** — 기능 A 완료 후 기능 B 시작, 문서 완성 후 다음 문서 |
| **신호 3** | **AI가 같은 실수를 반복할 때** → 컨텍스트 오염 신호. 더 대화해도 나아지지 않습니다 |

> 🗣 **강사 노트**

## SECTION 08 — 클로징 `1분`

```
AI는 이제 누구나 쓸 수 있는 도구입니다.
하지만 차이를 만드는 건 프롬프트가 아니라 컨텍스트,
그리고 그 컨텍스트를 지속 가능하게 만드는 건 하네스입니다.

기억은 압축됩니다. 기록만이 남습니다.
업무 매뉴얼을 만들고, 규칙을 세우고, 주기적으로 정리하세요.
AI의 업무 환경을 설계하는 사람이 AI를 제대로 쓰는 사람입니다.

같이 달리다 보면, 비로소 보입니다.
```

---

### 📌 오늘 배운 것 7가지

| | |
|---|---|
| 1 | 차이를 만드는 건 프롬프트 기술이 아니라, **내 경험과 판단이 담긴 맥락(컨텍스트)**이다. |
| 2 | AI는 받은 맥락을 증폭하는 도구다. 좋은 맥락 → 좋은 결과, 잘못된 맥락 → 그럴듯한 오답. |
| 3 | 컨텍스트 윈도우 = AI의 작업 메모리. 가득 차면 AI 품질이 저하된다. |
| 4 | **기억은 압축된다.** 결론만 남고 맥락은 사라진다. 기록이 있어야 AI에게 전달할 수 있다. |
| 5 | 새 에이전트는 기억이 없다. 그래서 하네스(업무 매뉴얼 + 가드레일 + 정기 점검)가 필요하다. |
| 6 | **AI 성능은 모델보다 환경이 결정한다.** 규칙이 명확할수록 AI는 더 정확하게 일한다. |
| **7** | **스킬 파일은 코드가 아닙니다. 누구든 텍스트로 만들어 쓸 수 있습니다.** |
