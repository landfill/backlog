# 에이전트 팀 구성 (Agent Roles)

## 개요

팀원 에이전트는 단일 에이전트가 아니라 **역할이 분화된 에이전트 팀**으로 구성된다.
각 에이전트는 고유한 역량을 가지며, Lead의 계획에 따라 파이프라인 또는 병렬로 협업한다.

## repository-level 역할 대응

이 문서는 app-level 역할을 설명하지만,
동시에 repository-level harness의 추상 역할을 앱 문맥으로 구체화한다.

| repository-level | common-employee app-level | 설명 |
|---|---|---|
| PM | Lead | 범위 고정, 완료 기준 확인, 최종 승인 |
| Coder | Analyst / Executor / Reporter | 조사, 실행, 기록 책임으로 분화 |
| Security Reviewer | Guardian | Gate 2 중심 보안 판정 |
| Tester | Guardian | Gate 1/2/3 품질 검증과 완료성 재확인 |

위 대응은 역할 이름을 바꾸기 위한 것이 아니라,
공통 pipeline 책임을 앱 문맥에 맞게 재배치한 것이다.

## 팀 구조

```
┌─────────────────────────────────────────────────────┐
│                  Lead (오케스트레이터)                   │
│          분류 · 계획 · 위임 · 최종 승인                  │
│          중간 매니저 필요 여부 동적 판단                   │
└──────────────────────┬──────────────────────────────┘
                       │
                       │ 프로젝트성 티켓일 때만 스폰
                       ▼
              ┌──────────────────┐
              │  Coordinator     │
              │  (조율자)         │
              │  일정 · 마일스톤  │
              └────────┬─────────┘
                       │
         ┌─────────────┼─────────────┬─────────────┐
         ▼             ▼             ▼             ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ Analyst  │  │ Executor │  │ Reporter │  │ Guardian │
   │ (분석가)  │  │ (실행자)  │  │ (보고관)  │  │ (검수관)  │
   └──────────┘  └──────────┘  └──────────┘  └──────────┘
```

## 역할 정의

### Lead (오케스트레이터)

팀 전체를 지휘하는 유일한 의사결정자.

**담당:**
- 티켓 수신 및 1차 분류 (운영/프로젝트/불명확)
- 작업 계획 수립 (작업 의존성 그래프 생성)
- 하위 에이전트에 작업 위임
- 병렬 실행 가능 여부 판단
- Coordinator 스폰 필요성 결정
- 최종 산출물 승인
- Human-in-the-loop 판단 (사람 개입이 필요한 시점 결정)

**위임하지 않는 것:**
- 티켓 분류 — Lead만 한다 (분류 오류는 이후 전체를 오염시킴)
- 사람에게 에스컬레이션 결정 — Lead만 한다
- 최종 승인 — Lead만 한다

**산출물:**
- 작업 계획서 (작업 DAG)
- 위임 지시
- 최종 승인 기록

---

### Coordinator (조율자) — 조건부 활성화

프로젝트성 티켓에서만 Lead가 스폰하는 중간 매니저.

**활성화 조건:**
- 티켓이 프로젝트성으로 분류된 경우
- 예상 처리 기간이 1주 이상인 경우
- 복수의 마일스톤이 필요한 경우

**담당:**
- 프로젝트 일정 수립 및 관리
- 마일스톤별 진행률 추적
- 하위 에이전트 작업 조율 (Lead의 계획에 따라)
- 정기 프로젝트 상태 보고 트리거
- 리스크/이슈 조기 감지 및 Lead에 보고

**위임받지 않는 것:**
- 티켓 분류 변경 (Lead 권한)
- 에스컬레이션 결정 (Lead 권한)
- 프로젝트 범위 변경 (Lead + 사람 승인 필요)

**비활성 시:**
- 운영 티켓은 Lead가 직접 하위 에이전트를 조율한다
- Coordinator가 없어도 파이프라인은 동일하게 작동한다

**산출물:**
- 프로젝트 일정표
- 마일스톤 체크 기록
- 리스크/이슈 목록
- `docs/exec-plans/active/` 아래 프로젝트 실행 계획 문서

---

### Analyst (분석가)

정보를 수집하고 구조화하는 역할. 조사와 정리를 통합 수행한다.

**담당:**
- 티켓 관련 자료 조사 (Confluence, Jira 이력, runbooks, past-solutions)
- 수집한 정보의 구조화 및 요약
- 매뉴얼 매칭 (어떤 runbook이 적용되는지 판단)
- 과거 유사 사례 검색 및 유사도 평가
- 솔루션 후보 목록 작성 (근거 포함)

**조사와 정리를 통합하는 이유:**
- 조사 중 부족한 정보를 즉시 추가 조사할 수 있다
- 정리하면서 발견한 갭을 바로 메울 수 있다
- 핸드오프 없이 "조사 → 발견 → 추가 조사 → 정리" 반복이 가능하다

**병렬 실행 가능:**
- 여러 소스를 동시에 조사할 수 있다
- Lead가 DAG에 병렬 조사 태스크를 배치하면 동시 실행한다

**산출물:**
- 조사 결과 보고서 (구조화된 형태)
- 솔루션 후보 목록 + 각 후보의 근거
- 매뉴얼 매칭 결과

---

### Executor (실행자)

확정된 솔루션을 실제로 수행하는 역할.

**담당:**
- 솔루션 실행 (Jira 티켓 상태 변경, 댓글 작성, 필드 업데이트)
- 외부 시스템 조작 (Confluence 문서 수정, 설정 변경 등)
- 실행 결과 수집 및 기록
- 실행 중 예상치 못한 상황 발생 시 즉시 Lead에 보고

**위임받지 않는 것:**
- 솔루션 선택 (Lead가 Analyst 결과를 보고 결정)
- 에스컬레이션 실행 (Lead 권한)

**실행 전 필수 조건:**
- Guardian의 Gate 1 (분석 품질) 통과 후에만 실행한다
- Lead 또는 Coordinator의 실행 승인이 있어야 한다

**산출물:**
- 실행 결과 기록 (무엇을 어디에 어떻게 적용했는지)
- 외부 시스템 변경 로그

---

### Reporter (보고관)

모든 문서 산출물을 작성하고 관리하는 역할.

**담당:**
- 의사결정 로그 작성 (decision-logging 스키마에 따라)
- 일별/주별 보고서 생성
- 유관자/팀 리더 대상 보고서 작성
- Confluence/Jira/Teams/Outlook에 실릴 보고 내용 초안 작성
- 보고서 템플릿 관리 및 개선
- 지식 베이스 업데이트 제안 문서화

**역할 경계:**
- Reporter는 보고 내용과 산출물의 작성 책임을 가진다.
- 외부 publish/send transport는 직접 수행하지 않고 Executor 또는 운영자에게 handoff한다.
- Outlook SMTP의 실제 발송은 운영자만 수행한다.

**보고서 + 템플릿을 통합하는 이유:**
- 보고서를 작성하는 에이전트가 템플릿의 문제점을 가장 잘 안다
- 템플릿 개선 → 보고서 품질 개선의 피드백 루프가 내부에서 완결된다

**산출물:**
- 의사결정 로그 (건별)
- 일별/주별 보고서
- 유관자 공유 문서
- 템플릿 개선 제안

---

### Guardian (검수관)

파이프라인의 품질 게이트를 운영하는 역할. 평가와 보안을 통합 수행한다.

**담당:**
- 파이프라인 각 단계의 품질 검증 (Gate 1, 2, 3)
- 보안 위반 사항 탐지
- 통과/반려 판정 및 반려 사유 기록
- 반복적 품질 이슈 패턴 감지 및 Lead에 보고

**게이트 체계:**

| 게이트 | 위치 | 검증 대상 | 주요 체크리스트 |
|---|---|---|---|
| **Gate 1** | Analyst → Executor | 분석 품질 | 근거 충분한가, 솔루션 후보가 타당한가, 누락된 조사 항목 없는가 |
| **Gate 2** | Executor → Reporter | 실행 결과 + 보안 | 실행이 정상 완료됐는가, 의도치 않은 변경 없는가, 보안 정책 위반 없는가 |
| **Gate 3** | Reporter → Lead | 보고서 품질 | 로그가 스키마를 준수하는가, 보고서가 템플릿에 부합하는가, 수신자 관점에서 명확한가 |

**evaluator + 보안을 통합하는 이유:**
- 보안은 별도 단계가 아니라 실행 결과 검증 시 함께 확인하는 것이 효율적
- Gate 2에서 "실행 결과가 정상인가"를 볼 때 보안 체크리스트를 함께 적용
- 보안 이슈는 파이프라인 끝이 아니라 실행 직후에 잡아야 피해를 최소화

**반려 시 행동:**
- 반려 사유와 개선 방향을 해당 에이전트에 전달
- 해당 에이전트가 보완 후 재제출
- 2회 연속 반려 시 Lead에 에스컬레이션

**산출물:**
- 게이트별 통과/반려 기록
- 반려 사유 및 개선 지시
- 품질 이슈 패턴 보고
- Gate 1/2/3별 review report (`docs/status/ongoing/`의 `*-review-report-guardian-gateN.md`)

## 에이전트 간 통신 규칙

### 통신 방식

모든 에이전트 간 통신은 **구조화된 마크다운 메시지**로 이루어진다.

```markdown
## 에이전트 메시지

- from: analyst
- to: guardian
- type: gate_request (gate_request | gate_result | task_assign | task_result | escalation)
- ticket: JIRA-1234
- timestamp: 2026-03-29T14:30:00
- content: (작업 결과 또는 요청 내용)
```

### 통신 경로 제약

```
Lead ←→ 모든 에이전트          (직접 지시/보고)
Coordinator ←→ 하위 에이전트    (Lead가 위임한 범위 내에서)
Analyst → Guardian             (Gate 1 요청)
Executor → Guardian            (Gate 2 요청)
Reporter → Guardian            (Gate 3 요청)
Guardian → 해당 에이전트        (통과/반려 결과)
```

**금지된 통신:**
- 하위 에이전트끼리 직접 작업을 위임하지 않는다 (반드시 Lead 또는 Coordinator를 통해)
- Guardian이 직접 작업을 수정하지 않는다 (반려 후 해당 에이전트가 수정)

## 작업 의존성 그래프 (DAG) 설계

Lead가 티켓을 받으면 작업 DAG를 생성한다.

### 운영 티켓 DAG 예시

```
[Lead: 분류]
     │
     ▼
[Analyst: 매뉴얼 조사] ──→ [Gate 1] ──→ [Executor: 실행] ──→ [Gate 2] ──→ [Reporter: 로그] ──→ [Gate 3] ──→ [Lead: 승인]
```

### 프로젝트 티켓 DAG 예시 (병렬 포함)

```
[Lead: 분류]
     │
     ▼
[Coordinator: 일정 수립]
     │
     ├──→ [Analyst: 요구사항 분석]  ─┐
     ├──→ [Analyst: 기술 조사]      ─┼──→ [Analyst: 결과 종합] ──→ [Gate 1]
     └──→ [Analyst: 유관자 조사]    ─┘           │
                                                 ▼
                                     [Executor: 1단계 실행] ──→ [Gate 2]
                                                 │
                                                 ▼
                                     [Reporter: 중간 보고] ──→ [Gate 3]
                                                 │
                                                 ▼
                                     [Coordinator: 마일스톤 체크]
                                                 │
                                                 ▼
                                            (다음 단계...)
```

### 병렬 실행 판단 기준

Lead가 DAG 생성 시 적용하는 규칙:

1. **데이터 독립성**: 서로 다른 소스를 조사하는 작업은 병렬 가능
2. **순서 의존성**: 이전 작업의 결과가 다음 작업의 입력이면 직렬
3. **게이트 의존성**: Gate를 통과해야 다음 단계로 진행
4. **자원 충돌**: 같은 외부 시스템을 동시에 수정하면 직렬 (읽기는 병렬 가능)

## Coordinator 스폰 판단 기준

Lead가 다음 조건 중 하나 이상을 만족하면 Coordinator를 활성화한다:

| 조건 | 근거 |
|---|---|
| 티켓이 프로젝트성으로 분류됨 | 일정/마일스톤 관리 필요 |
| 예상 처리 기간 1주 이상 | 단기 처리가 아니므로 진행 추적 필요 |
| 하위 작업이 3개 이상 | 조율 복잡도가 Lead 직접 관리의 한계 초과 |
| 정기 보고가 2회 이상 필요 | 반복적 상태 점검이 필요 |

**스폰하지 않는 경우:**
- 단순 운영 티켓 (분류 → 매뉴얼 매칭 → 실행으로 끝남)
- 1일 이내 처리 가능한 티켓
- 하위 작업이 선형 파이프라인으로 충분한 경우

## 공유 자원과 개별 상태

### 공유 (모든 에이전트 접근 가능)
- `docs/domain-knowledge/` — 지식 베이스 (읽기 전용, 업데이트는 Reporter가 제안)
- `docs/templates/` — 보고서 템플릿 (읽기 전용, 업데이트는 Reporter가 관리)
- `docs/generated/decision-logs/` — 의사결정 로그 (Reporter가 작성, 전원 읽기 가능)

### 개별 (해당 에이전트만 쓰기 가능)
- 작업 결과물 — 각 에이전트가 자기 산출물만 작성
- 게이트 판정 — Guardian만 작성
- 작업 계획 — Lead (또는 위임 시 Coordinator)만 작성
