# 지식 정제 파이프라인 (Knowledge Pipeline)

## 목적

비정형 원본 자료(위키, PPT, Word, 과거 티켓)를 에이전트가 활용할 수 있는 **구조화된 runbook과 past-solution으로 변환**하는 파이프라인을 정의한다.

---

## 파이프라인 개요

```
[입력] 원본 자료
  │
  ├─ API 접근 가능 (Confluence) → Analyst 자동 수집
  └─ API 불가 (PPT, Word, PDF) → 사용자 제공 → raw/ 저장
  │
  ▼
[1단계] 수집 + 등록
  │  Analyst: raw/에 저장, _intake-log.md에 등록
  ▼
[2단계] 분류
  │  Analyst: 원본이 runbook인지, past-solution인지, 기타인지 판별
  ▼
[3단계] 구조화 정제
  │  Analyst: 표준 템플릿에 맞게 변환
  ▼
[4단계] 품질 검증
  │  Guardian: 정제 품질 검증
  ▼
[5단계] 배치
  │  Reporter: runbooks/ 또는 past-solutions/에 배치
  ▼
[6단계] 활용 + 피드백
  │  전체 에이전트: 티켓 처리 시 참조 → 정확도 추적
  ▼
[7단계] 최신화
     Analyst: 원본 변경 감지 → 재정제
```

---

## 1단계: 수집 + 등록

### API 수집 (Confluence)

```
[Analyst] Confluence CQL 검색
  │  label IN ("runbook", "manual", "guide", "howto")
  │  OR space IN (지정된 지식 스페이스)
  ▼
[Analyst] 페이지 본문 추출 (HTML → 마크다운)
  ▼
[Analyst] raw/ 에 저장
  ▼
[Analyst] _intake-log.md 에 등록
```

### 사용자 제공 (PPT, Word, PDF)

```
사용자가 파일 제공
  ▼
[Analyst] 텍스트 추출
  ├─ PPT → 슬라이드별 텍스트 + 노트
  ├─ Word → 본문 + 스타일 구조
  └─ PDF → OCR 또는 텍스트 추출
  ▼
[Analyst] 마크다운으로 변환 → raw/ 저장
  ▼
[Analyst] _intake-log.md 에 등록
```

### _intake-log.md 등록 형식

```markdown
### 2026-03-29 — 시스템A 장애 대응 매뉴얼
- **출처**: Confluence OPS 스페이스 / page-id: 12345
- **형태**: 위키
- **수집 방법**: API 추출
- **정제 상태**: 미정제
- **정제 결과 위치**: (정제 후 기입)
- **비고**: 없음
```

---

## 2단계: 분류

Analyst가 원본 내용을 분석하여 유형을 판별한다.

| 유형 | 특징 | 정제 대상 |
|---|---|---|
| **매뉴얼/가이드** | 절차가 있음, "~하세요" 형태, 단계별 설명 | → runbooks/ |
| **과거 사례** | 특정 문제 해결 기록, 티켓 번호 포함, 원인-해결 구조 | → past-solutions/ |
| **참고 자료** | 아키텍처 문서, 설계서, 정책 문서 | → references/ (정제 대상 아님, 참조용 보관) |
| **복합** | 매뉴얼 + 사례가 혼재 | → 분리하여 각각 정제 |

---

## 3단계: 구조화 정제

### Runbook 정제 규칙

원본에서 아래 구조를 추출/생성한다:

```markdown
# [매뉴얼 제목]

## 메타 정보
- 원본 출처: (raw/ 파일 경로 또는 Confluence URL)
- 정제일: YYYY-MM-DD
- 정제자: Analyst (AI)
- 검수자: (사람 이름 또는 "미검수")
- last_verified: YYYY-MM-DD
- 참조 횟수: 0
- 성공률: N/A

## 적용 조건
- 이 매뉴얼이 적용되는 상황 (티켓 유형, 시스템, 증상 등)
- 적용하면 안 되는 상황 (예외 조건)

## 절차
1. [단계 1 설명]
2. [단계 2 설명]
3. ...

## 예외 처리
- [예외 상황 1]: [대응 방법]
- [예외 상황 2]: [대응 방법]

## 관련 자료
- [다른 runbook 또는 참고 문서 링크]
```

### Past Solution 정제 규칙

```markdown
# [사례 제목]

## 메타 정보
- 원본 티켓: JIRA-XXXX (또는 원본 출처)
- 발생일: YYYY-MM-DD
- 정제일: YYYY-MM-DD
- 정제자: Analyst (AI)
- last_verified: YYYY-MM-DD
- 재사용 횟수: 0
- 신뢰도: 초기

## 증상
- 사용자가 보고한 증상 / 에러 메시지 / 현상

## 원인
- 근본 원인 분석 결과

## 해결 방법
1. [단계 1]
2. [단계 2]
3. ...

## 적용 조건
- 이 솔루션이 유효한 조건
- 적용하면 안 되는 조건

## 교훈
- 이 사례에서 배운 점, 재발 방지 조치
```

### 정제 시 주의사항

- 원본의 의미를 변경하지 않는다
- 불확실한 부분은 `[확인 필요]` 태그를 남긴다
- 원본에 없는 내용을 추가하지 않는다 (AI 추측 금지)
- 절차의 순서가 불명확하면 원본 그대로 옮기고 `[순서 확인 필요]` 표시

---

## 4단계: 품질 검증

Guardian이 정제 결과를 검증한다.

| 항목 | 기준 |
|---|---|
| 구조 준수 | 템플릿의 모든 섹션이 있는가 |
| 원본 충실성 | 원본 내용과 불일치하는 부분이 없는가 |
| 적용 조건 명확성 | "언제 쓰는 매뉴얼인지"가 명확한가 |
| 절차 실행 가능성 | 절차대로 따라할 수 있는 수준인가 |
| 불확실 표시 | `[확인 필요]` 태그가 적절히 사용되었는가 |

반려 시 Analyst가 보완 후 재제출.

---

## 5단계: 배치

Reporter가 검증 통과한 문서를 최종 위치에 배치한다.

### Runbook 배치 규칙

- 티켓 유형이 명확 → `runbooks/by-ticket-type/[유형].md`
- 시스템이 명확 → `runbooks/by-system/[시스템].md`
- 둘 다 해당 → 주 기준으로 배치 + 다른 경로에 링크

### Past Solution 배치 규칙

- 파일명: `past-solutions/YYYY-MM-DD-TICKET-ID.md` (원본 티켓 기준)
- 원본 티켓이 없는 경우: `past-solutions/YYYY-MM-DD-[간략제목].md`

---

## 6단계: 활용 + 피드백

### 참조 추적

Analyst가 솔루션 탐색 시 매뉴얼을 참조하면:
- 해당 매뉴얼의 `참조 횟수` +1
- 처리 결과가 성공이면 `성공률` 업데이트

### Past Solution 신뢰도

| 재사용 횟수 | 성공률 | 신뢰도 |
|---|---|---|
| 0회 | N/A | 초기 |
| 1~2회 | 100% | 중간 |
| 3회 이상 | ≥80% | 높음 |
| 아무 횟수 | <50% | 낮음 (재검토 필요) |

---

## 7단계: 최신화

### 자동 감지

- `last_verified`가 3개월 이상 경과한 문서 → 재검증 대상 표시
- 원본 Confluence 페이지가 수정된 경우 → 재정제 트리거
- 신뢰도 "낮음"인 문서 → 재검토 대상

### 재정제 프로세스

```
[트리거] 최신화 필요 감지
  ↓
[Analyst] 원본 재확인 (변경점 파악)
  ↓
[Analyst] 기존 정제본 업데이트
  ↓
[Guardian] 품질 재검증
  ↓
[Reporter] last_verified 갱신
```
