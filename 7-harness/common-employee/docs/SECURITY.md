# common-employee Security

## 목적

이 문서는 `common-employee` 앱이 Jira, Confluence, Teams, Outlook과 연동될 때
지켜야 하는 앱 레벨 보안 기준을 정의한다.

repository-level `harness/core/docs/repository-security.md`를
이 앱 문맥으로 더 구체화한 문서다.

## 기본 원칙

- 보호 대상 데이터는 기본적으로 저장하지 않는다.
- 저장이 필요하면 최소 범위만 저장하고 이유를 남긴다.
- 읽기와 쓰기 권한은 역할 기준으로 분리한다.
- 보안이 불분명하면 자율 처리하지 않고 멈춘다.

## 보호 대상 분류

| 등급 | 예시 | 기본 처리 |
|---|---|---|
| Restricted | API 키, OAuth 토큰, 비밀번호, 클라이언트 시크릿, 세션 정보 | 문서/로그 저장 금지, 환경 변수 또는 시크릿 매니저만 허용 |
| Sensitive | 메일 본문, 채팅 내용, 첨부파일, 개인정보가 포함된 티켓 내용, 내부 운영 절차 중 비공개 정보 | 최소 인용만 허용, 원문 복제 금지, 필요 시 마스킹 후 제한 저장 |
| Internal | 티켓 메타데이터, runbook 참조 정보, 의사결정 근거 요약, 게이트 판정 | 업무 범위 안에서 문서화 가능, 외부 공유 시 최소화 |
| Public | 외부 공개가 허용된 안내 문구 또는 승인된 템플릿 | 명시적으로 공개 허용된 경우만 사용 |

`Public`으로 분류되지 않은 데이터는 기본적으로 `Internal` 이상으로 취급한다.

## 신뢰 경계

이 앱은 아래 경계를 넘을 때마다 데이터 최소화와 권한 검증을 수행한다.

- 외부 시스템 ↔ 에이전트 실행 환경
- 로컬 브라우저 ↔ 로컬 웹 콘솔
- 에이전트 실행 환경 ↔ 문서 저장소
- 내부 상태 문서 ↔ 외부 공유 채널
- 사람 승인 전 ↔ 사람 승인 후

## V1 보안 결정

### 1. 서비스 계정 원칙

- Jira, Confluence, Teams, Outlook 연동은 개인 계정이 아니라 서비스 계정만 사용한다.
- 서비스 계정 권한은 `docs/integrations/auth-strategy.md`의 least privilege 표를 기준으로 한다.
- 개발/검증/운영 환경은 자격 증명을 분리한다.

### 2. 역할 기반 권한 분리

| 역할 | 허용 | 금지 |
|---|---|---|
| Lead | 분류, 승인, 사람 개입 결정 | 자격 증명 저장, 비승인 외부 쓰기 |
| Analyst | 읽기 중심 조사, 근거 수집 | 외부 시스템 쓰기, 민감 원문 무단 복제 |
| Executor | 승인된 범위 안의 외부 쓰기 | Gate 1 전 실행, 범위 밖 수정 |
| Reporter | 요약 로그/보고 작성 | 민감 원문 전체 복사 |
| Guardian | 보안/품질 판정 | 직접 수정, 보안 의심 사항 무시 |
| Coordinator | 일정/마일스톤 조율 | 권한 확장 승인 |

### 3. 원문 저장 제한

- Jira/Teams/Outlook의 원문 본문과 첨부파일은 기본적으로 장기 저장하지 않는다.
- 지식 정제를 위해 원본 저장이 꼭 필요하면 `docs/domain-knowledge/raw/`에 제한 저장하고,
  intake 기록에 저장 이유와 출처를 남긴다.
- decision log와 보고서에는 원문 전체 대신 요약, 참조 경로, 마스킹된 발췌만 남긴다.

### 4. 외부 공유 최소화

- Teams, Outlook, Jira 댓글에는 처리 결과와 필요한 근거만 보낸다.
- 민감 정보, 내부 토큰, 전체 첨부 내용, 내부 운영 세부 절차는 외부 공유 메시지에 넣지 않는다.
- 링크 공유가 가능한 경우 본문 복제보다 링크를 우선한다.

### 5. 로컬 웹 콘솔 노출 제한

- 웹 콘솔은 로컬 개발/검증 환경 전용이다.
- 웹 콘솔은 `docs/status/*`와 `docs/generated/decision-logs/*`의 generated artifact만 열람 대상으로 허용한다.
- 워크스페이스 밖 경로, DB 원시 내용, redaction 전 민감 본문은 브라우저에 직접 노출하지 않는다.
- 브라우저에서 제출하는 intake payload도 기존 runtime redaction 규칙과 동일한 기준을 따른다.
- Jira Cloud 자격 증명은 환경 변수로만 주입하고 UI에 노출하지 않는다.

## 허용 / 금지 데이터 흐름

| 데이터 흐름 | 허용 여부 | 조건 |
|---|---|---|
| 티켓 메타데이터 → tracker / ongoing plan | 허용 | 티켓 ID, 상태, owner, 다음 행동 수준만 기록 |
| 티켓 메타데이터 → `runtime/autonomous-runtime.db` | 허용 | stage, verdict, attempts, redacted evidence reference만 저장 |
| runbook 경로 / 사례 ID → decision log | 허용 | 참조 근거 수준으로만 기록 |
| 민감 원문 → decision log / 보고서 | 금지 | 요약 또는 마스킹된 발췌만 허용 |
| 민감 원문 → `runtime/autonomous-runtime.db` | 금지 | redacted summary 없이 원문 저장 금지 |
| 자격 증명 → 코드 / 문서 / 로그 | 금지 | 환경 변수 또는 시크릿 매니저만 허용 |
| 민감 데이터 → LLM 입력 | 조건부 허용 | 목적 최소화, 직접 식별자 마스킹, 불필요 원문 제거 |
| 원본 첨부파일 → `docs/domain-knowledge/raw/` | 조건부 허용 | intake 기록, 저장 이유, 접근 범위가 남아야 함 |
| 외부 시스템 결과 → 외부 공유 채널 | 조건부 허용 | 유관자에게 필요한 최소 결과만 전달 |
| generated artifact → 로컬 웹 콘솔 | 허용 | `docs/status/*`, `docs/generated/decision-logs/*` 하위의 redacted/generated 파일만 허용 |
| 워크스페이스 밖 파일 → 로컬 웹 콘솔 | 금지 | 경로 탐색/원시 파일 노출 금지 |
| Jira issue 본문 → 로컬 artifact | 조건부 허용 | 평탄화/요약 후 최소 범위만 허용, 원문 전체 복제 금지 |

## 인증 / 자격 증명 규칙

- 자격 증명 저장 방식은 `docs/integrations/auth-strategy.md`를 따른다.
- `.env`, 예시 코드, 스크린샷, 로그에 실제 비밀값을 남기지 않는다.
- 자격 증명 누출이 의심되면 해당 토큰을 즉시 무효화 후보로 표시하고 `BLOCKED`로 전환한다.
- 로컬 테스트에서도 운영 자격 증명을 재사용하지 않는다.

## Gate 2 보안 체크리스트

Guardian은 실행 결과 검증 시 최소한 아래를 확인한다.

- 외부 쓰기 대상이 승인된 시스템과 리소스 범위 안에 있는가
- 실행 로그나 보고서에 민감 정보가 그대로 남지 않았는가
- 서비스 계정 권한을 넘어서는 액션이 시도되지 않았는가
- 원문 복제가 필요한 경우 저장 이유와 경로가 남아 있는가
- 의도하지 않은 부가 변경이 없는가
- 다음 단계가 이어받을 때 민감 정보가 과도하게 노출되지 않는가

## 차단 조건

아래 중 하나라도 발생하면 즉시 `BLOCKED` 처리한다.

- 자격 증명 또는 시크릿 값 노출
- 범위 밖 외부 시스템 쓰기
- 민감 데이터의 무단 저장 또는 무단 외부 전송
- 승인되지 않은 사람 개입 우회
- 어떤 데이터가 보호 대상인지 판별할 수 없는 상태

## 사고 대응

### 시크릿 노출 의심

1. 추가 자동 처리를 중단한다.
2. 노출 위치와 범위를 기록하되 비밀값 자체는 남기지 않는다.
3. Lead에 즉시 보고하고 사람 승인 없이는 재개하지 않는다.
4. 토큰 폐기, 회전, 영향 범위 확인을 후속 액션으로 남긴다.

### 의도치 않은 외부 쓰기

1. 실제 반영 범위를 확인한다.
2. 되돌릴 수 있는 경우 복구 계획을 먼저 적는다.
3. Guardian이 보안 영향도를 판정한다.
4. 사용자 또는 보안 담당자 승인 전까지 후속 자동 실행을 멈춘다.

## 감사와 추적성

- 모든 외부 쓰기에는 대상 시스템, 액션, 결과가 남아야 한다.
- 모든 보안 관련 반려에는 이유와 경로가 남아야 한다.
- 월별/분기별 권한 점검 기준은 `docs/integrations/auth-strategy.md`의 감사 항목을 따른다.
