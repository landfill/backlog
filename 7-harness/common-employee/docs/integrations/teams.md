# Teams 연동 설계

## 역할

Teams는 **진행 상황을 운영자 자신에게 알리는 셀프 알림 채널**이다.
현재 범위에서는 고정 Incoming Webhook으로만 사용하며,
사용자 개입 요청, DM, 응답 수신, 채널 탐색은 수행하지 않는다.

## 현재 기준선

- Teams 연동 기준선은 **고정 웹훅 단방향 발송**이다.
- 목적은 처리 시작, 지연, 완료, 오류 같은 진행 상태를 운영자가 빠르게 인지하는 것이다.
- Teams는 사람의 답변을 받는 채널이 아니라 상태 알림 채널이다.
- Graph 기반 Teams 권한, 채팅 생성, Adaptive Card 응답 모델은 현재 기준선에서 제외한다.

## 역할별 관여 범위

| 주체 | 역할 | 주요 용도 |
|---|---|---|
| Executor | 발송 | 진행 상황 셀프 알림 |
| Lead | 읽기 | 운영 상태 인지 |

---

## 메시지 유형

### 1. 처리 진행 알림

**발송 시점:**
- 처리 시작
- 중간 진행
- 처리 완료
- 지연 예상
- 오류 또는 차단 발생

```markdown
✅ [JIRA-1234] 처리 완료

runbook 4.1절 절차 적용, 서비스 정상 확인.
상세: [Jira 티켓 링크]
```

### 2. 주기 보고 링크 공유

일별/주별 보고서 생성 후 운영자 확인용 링크를 공유할 수 있다.

### 3. 차단/오류 알림

```markdown
🚨 [JIRA-1234] 처리 차단

원인: SMTP 인증 실패
다음 조치: 운영자 설정 확인 필요
상세: [Jira 티켓 링크]
```

---

## 채널 전략

| 메시지 유형 | 채널 | 이유 |
|---|---|---|
| 진행 알림 | 고정 운영 채널 | 운영자 가시성 |
| 보고 링크 공유 | 고정 운영 채널 | 주기 점검 |
| 오류/차단 알림 | 고정 운영 채널 | 즉시 인지 |

**채널 설정:**

```yaml
channels:
  progress_webhook: ""  # 진행 상황 셀프 알림용 Incoming Webhook URL
```

---

## 제외 범위

- 1:1 DM
- 사용자 응답 수신
- Adaptive Card
- 채널 목록 조회
- 멘션/개별 대상자 식별
- Teams를 통한 사람 승인 수집

---

## 메시지 형식 규칙

- 모든 메시지에 `[🤖 팀원에이전트]` 식별자 포함
- 티켓 번호를 항상 포함하여 추적 가능하게
- 한 메시지에 하나의 주제만 담는다
- 메시지 길이: 최대 500자
- 외부 공유용이 아니라 운영자 셀프 알림임을 전제로 간결하게 유지한다

---

## 에러 처리

| 에러 | 재시도 | 대응 |
|---|---|---|
| 401/403/404 | ❌ | 웹훅 설정 오류로 간주하고 Lead 보고 |
| 429 | ✅ 3회 | Retry-After 준수 |
| 발송 실패 | ✅ 2회 | 최종 실패 시 Jira 댓글 또는 운영 로그에 기록 |
