# Outlook 연동 설계

## 역할

Outlook은 **관리자 수동 발송이 필요한 공식 메일 채널**이다.
자동 메일함 연동이나 응답 수집 채널이 아니라,
운영자가 UI에서 수신자를 직접 지정하고 발송 버튼을 눌렀을 때만 사용하는 보조 채널이다.

## 현재 기준선

- 현재 Outlook 연동 기준선은 **SMTP 기반 수동 발송**이다.
- 운영자가 UI에서 제목/본문/수신자를 확인하고 직접 발송 액션을 수행한다.
- 메일 읽기, 수신함 모니터링, 답장 자동 수집은 현재 범위에 포함되지 않는다.
- Graph 기반 Outlook 권한 모델은 현재 기준선에서 제외한다.

## 역할별 관여 범위

| 주체 | 역할 | 주요 용도 |
|---|---|---|
| Executor | 발송 초안 준비 | 제목/본문/발송 이유를 준비 |
| 운영자(관리자) | 실제 발송 실행 | UI에서 수신자 지정 후 SMTP 발송 |
| Lead | 발송 승인/판단 | 긴급도와 발송 필요성 판단 |

---

## 발송 시나리오

Outlook은 모든 티켓에서 자동으로 쓰지 않는다.
아래와 같이 운영자가 직접 보내는 경우만 허용한다.

| 조건 | 설명 |
|---|---|
| 긴급 에스컬레이션 | Jira 댓글만으로는 즉시 확인이 어려운 경우 |
| 공식 보고 | 주별 보고 등 메일 기록이 필요한 경우 |
| 외부 유관자 | Jira 접근이 없거나 Teams 웹훅 알림 대상이 아닌 경우 |

### 발송하지 않는 경우

- 일반 처리 알림
- 분류 확인/응답 수집 목적의 자동 메일
- 수신함 감시를 전제로 한 후속 자동화

---

## 발송 규칙

- 수신자는 UI에서 운영자가 직접 입력하거나 검토한다.
- 시스템은 기본 발신 계정 정보와 메일 초안을 제공할 수 있지만, 실제 전송은 운영자 액션이 있어야 한다.
- 메일 본문은 HTML 또는 plain text 중 구현 시점에 단일 형식으로 고정한다.
- 자동 발송처럼 오해되지 않도록 서명에 "운영자 확인 후 발송" 또는 동등한 고지를 포함한다.

### 공통 형식

- 제목: `[팀원에이전트] {유형} — {티켓번호} {요약}`
- 본문: 간결하고 사실 중심
- 링크: Jira 티켓 링크 우선

---

## 수신 처리

- 수신 메일 모니터링은 현재 범위에서 제외한다.
- 메일 답장이나 회신은 시스템이 자동으로 해석하지 않는다.
- 메일 기반 후속 조치가 필요하면 운영자가 Jira에 수동 반영한다.

---

## 에러 처리

| 에러 | 재시도 | 대응 |
|---|---|---|
| SMTP 인증 실패 | ❌ | 설정 점검 후 Lead 보고 |
| SMTP 연결 실패 | ✅ 2회 | 연속 실패 시 발송 중단 + Lead 보고 |
| 발송 실패 | ✅ 2회 | 최종 실패 시 Jira 댓글로 기록 + 운영자 재시도 판단 |
