# Outlook 연동 설계

## 역할

Outlook은 **공식 알림과 에스컬레이션 채널**이다.
Teams의 폴백 채널이기도 하며, 기록이 남아야 하는 공식 커뮤니케이션에 사용한다.

## 사용 에이전트별 권한

| 에이전트 | 읽기 | 쓰기 | 주요 용도 |
|---|---|---|---|
| Executor | ✅ | ✅ | 메일 발송, 수신 메일 확인 |
| Lead | ✅ | ❌ | 사용자 응답 확인 |

---

## 발송 시나리오

Outlook은 Teams와 달리 **선택적으로 사용**한다. 모든 건에 메일을 보내지 않는다.

### 발송 조건

| 조건 | 설명 |
|---|---|
| Teams 장애 | Teams 불가용 시 폴백으로 발송 |
| 긴급 에스컬레이션 | Teams + Outlook 동시 발송 (이중 알림) |
| 공식 보고 | 주별 보고서 등 공식 기록이 필요한 경우 |
| 외부 유관자 | Teams에 없는 외부 유관자에게 알림 |

### 발송하지 않는 경우

- 일반 처리 알림 (Teams로 충분)
- 일상적 개입 요청 (Teams DM 사용)
- 리마인더 (Teams 사용)

## 메일 형식

### 공통 규칙

- 발신자: 에이전트 전용 메일 주소 (예: `teammate-agent@company.com`)
- 제목: `[팀원에이전트] {유형} — {티켓번호} {요약}`
- 본문: HTML 형식, 간결하게
- 서명: 에이전트 식별 + 자동 생성 고지

### 긴급 에스컬레이션 메일

```
제목: [팀원에이전트] 🚨 긴급 — JIRA-1234 보안 이슈 감지

본문:
━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚨 긴급 확인 요청
━━━━━━━━━━━━━━━━━━━━━━━━━━━

티켓: JIRA-1234
상황: Gate 2 검증 중 보안 위반 의심 사항 감지
영향 범위: 시스템A
현재 상태: 처리 중단, 사람 확인 대기 중

상세: [Jira 티켓 링크]

━━━━━━━━━━━━━━━━━━━━━━━━━━━
이 메일은 팀원 에이전트가 자동으로 발송했습니다.
```

### 주별 보고 메일

```
제목: [팀원에이전트] 📊 주별 보고 — 2026-W13

본문:
주간 처리 현황 요약

- 처리 건수: 15건 (자율 12건 / 개입 3건)
- 자율 처리율: 80%
- 평균 처리 시간: 35분
- Guardian 반려: 2건

상세 보고서: [Confluence 링크]

━━━━━━━━━━━━━━━━━━━━━━━━━━━
이 메일은 팀원 에이전트가 자동으로 발송했습니다.
```

## 수신 처리

### 수신 메일 모니터링

에이전트 전용 메일함으로 수신되는 메일을 모니터링한다.

**모니터링 대상:**
- 에스컬레이션에 대한 답장
- 보고서에 대한 피드백
- 사용자가 직접 메일로 보낸 요청

**모니터링 방식**: 주기적 폴링 (5분 간격)

**처리 플로우:**

```
[Executor] 새 메일 감지
  ↓
[Lead] 메일 내용 해석
  ├─ 티켓 관련 응답 → 해당 티켓 파이프라인 재개
  ├─ 신규 요청 → Jira 티켓 생성 검토
  └─ 해석 불가 → 원문 보존 + 팀 리더에 전달
```

---

## 수신자 목록 관리

```yaml
recipients:
  team_leader: ""          # 팀 리더 메일 주소
  security_contact: ""     # 보안 담당자
  distribution_list: ""    # 보고서 수신 그룹
```

> ⚠️ 실제 메일 주소는 Phase 4에서 구체화한다.

---

## 에러 처리

| 에러 | 재시도 | 대응 |
|---|---|---|
| 401 Unauthorized | ❌ | 토큰 만료 → Lead 보고 |
| 429 Throttled | ✅ 3회 | Retry-After 준수 |
| 발송 실패 | ✅ 2회 | 최종 실패 시 Jira 댓글로 기록 + Lead 보고 |
| 수신함 접근 불가 | ✅ 3회 | 연속 실패 시 Lead 보고 |
