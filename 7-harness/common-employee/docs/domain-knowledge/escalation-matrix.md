# 에스컬레이션 매트릭스 (Escalation Matrix)

## 목적

에이전트가 자율 처리할 수 없는 상황에서 **누구에게, 어떤 방식으로 에스컬레이션하는지** 정의한다.
에스컬레이션 대상이 모호한 채로 에스컬레이션하지 않는다 (AGENTS.md 제약).

---

## 에스컬레이션 판단 주체

- **판단**: Lead (전용 권한)
- **실행**: Executor (Teams/Outlook/Jira)
- **기록**: Reporter (의사결정 로그)

---

## 에스컬레이션 트리거

| 트리거 | 긴급도 | 설명 |
|---|---|---|
| 솔루션 탐색 실패 + 확신도 낮음 | 보통 | 매뉴얼/사례 없음, 사람 판단 필요 |
| Guardian 2회 연속 반려 | 보통 | 에이전트 자체 해결 불가 |
| 보안 위반 의심 | 긴급 | 즉시 사람 확인 필요 |
| SLA 70% 도달 | 보통 | 처리 지연 위험 |
| 사용자 개입 타임아웃 반복 | 보통 | 사용자 무응답 지속 |
| 시스템 장애 확산 위험 | 긴급 | 즉시 사람 확인 필요 |

## 에스컬레이션 대상

> ⚠️ 아래 표의 실제 이름/연락처는 운영 시 구체화한다.

| 대상 | 역할 | 에스컬레이션 상황 | 연락 방식 |
|---|---|---|---|
| 팀 리더 | 최종 판단권자 | 모든 에스컬레이션의 기본 대상 | Teams DM → Outlook |
| 보안 담당자 | 보안 이슈 전문 | 보안 위반 의심 | Teams DM + Outlook 동시 |
| 시스템 담당자 | 특정 시스템 전문 | 해당 시스템 관련 기술 이슈 | Teams DM |
| 기획 담당자 | 요구사항 판단 | 프로젝트성 티켓의 범위/요구사항 모호 | Teams DM |

### 대상 선택 규칙

```
에스컬레이션 필요
  │
  ├─ 보안 관련 → 보안 담당자 + 팀 리더
  ├─ 특정 시스템 관련 → 시스템 담당자 (불명 시 팀 리더)
  ├─ 요구사항 관련 → 기획 담당자 (불명 시 팀 리더)
  └─ 그 외 모든 경우 → 팀 리더
```

## 에스컬레이션 형식

### Jira

```markdown
[🤖 팀원에이전트] 에스컬레이션

**사유**: [에스컬레이션 이유]
**시도한 것**: [에이전트가 수행한 조사/처리 내역 요약]
**필요한 판단**: [사람이 결정해야 할 것]
**판단 근거 상세**: [의사결정 로그 링크]
```

### Teams (긴급)

```markdown
🚨 [JIRA-1234] 에스컬레이션

사유: [간결한 설명]
시도한 것: [요약]
필요한 조치: [구체적으로]

상세: [Jira 티켓 링크]
```

## 에스컬레이션 후 행동

| 상황 | 에이전트 행동 |
|---|---|
| 에스컬레이션 전달 완료 | 티켓 상태 "보류"로 변경, 대기 |
| 사람이 해결 | 결과를 의사결정 로그에 기록, 학습 |
| 사람이 에이전트에 재지시 | 새로운 지시에 따라 파이프라인 재개 |
| 응답 없음 (8시간) | 리마인더 1회 → 상위 에스컬레이션 (팀 리더) |

## 에스컬레이션 추적

Reporter가 주별 보고에 포함하는 에스컬레이션 지표:

| 지표 | 설명 |
|---|---|
| 에스컬레이션 건수 | 주간 총 건수 |
| 에스컬레이션 사유별 분류 | 어떤 이유로 에스컬레이션했는가 |
| 평균 응답 시간 | 에스컬레이션 → 사람 응답까지 소요 시간 |
| 반복 에스컬레이션 | 같은 유형이 반복되면 매뉴얼 보강 필요 |
