# 솔루션 탐색 (Solution Lookup)

## 목적

분류된 티켓에 대해 **적용 가능한 솔루션 후보를 탐색하고 근거와 함께 제출**한다.
Lead가 최종 솔루션을 선택할 수 있도록 충분한 정보를 구조화하여 제공한다.

## 담당 에이전트

- **탐색 실행**: Analyst
- **탐색 범위/우선순위 지시**: Lead (작업 DAG에 포함)
- **탐색 결과 검증**: Guardian (Gate 1)

---

## 탐색 소스와 우선순위

Analyst는 아래 순서로 탐색한다. 앞선 소스에서 확신도 높은 매칭이 나오면 뒤 소스는 보조 확인용으로만 사용한다.

| 순위 | 소스 | 위치 | 이유 |
|---|---|---|---|
| 1 | **Runbooks** | `docs/domain-knowledge/runbooks/` | 검증된 공식 매뉴얼, 가장 신뢰도 높음 |
| 2 | **Past Solutions** | `docs/domain-knowledge/past-solutions/` | 실제 처리 성공 사례, 실전 검증됨 |
| 3 | **Confluence 위키** | 외부 시스템 | 팀 공유 지식, 매뉴얼 외 참고 자료 |
| 4 | **Jira 이력** | 외부 시스템 | 유사 티켓의 댓글/해결 내역 |

### 소스별 탐색 방법

**Runbooks 탐색:**
1. 티켓 유형 기반: `runbooks/by-ticket-type/` 에서 이슈유형 매칭
2. 시스템 기반: `runbooks/by-system/` 에서 관련 시스템 매칭
3. 키워드 기반: 티켓 제목/설명의 핵심 키워드로 전체 runbooks 검색
4. 매칭도 판정: 정확히 매칭 / 부분 매칭 / 매칭 없음

**Past Solutions 탐색:**
1. 증상 유사도: 티켓의 증상/에러와 과거 사례의 증상 비교
2. 시스템 동일성: 같은 시스템에서 발생한 사례 우선
3. 최신성: 최근 사례일수록 가중치 부여
4. 성공률: 해당 솔루션의 과거 적용 성공 횟수

**Confluence 검색:**
- 키워드 기반 검색 + 관련 스페이스 제한
- 검색 결과에서 절차/가이드 형태의 페이지 우선

**Jira 이력 검색:**
- 유사 제목/설명의 해결된 티켓 검색
- 해결 댓글에서 실제 적용된 솔루션 추출
- 동일 보고자/시스템의 이전 티켓 확인

## 탐색 실행 모델

### 단순 운영 티켓 (직렬)

```
[Analyst] Runbooks 검색
  │
  ├─ 정확히 매칭 → 솔루션 후보 확정 + Past Solutions에서 보조 확인
  │
  └─ 매칭 없음/부분 매칭 → Past Solutions 검색 → Confluence → Jira 이력
```

### 복합 티켓 또는 프로젝트성 (병렬)

Lead가 DAG에 병렬 탐색을 배치하면:

```
┌─→ [Analyst] Runbooks 검색          ─┐
├─→ [Analyst] Past Solutions 검색     ─┼─→ [Analyst] 결과 종합
├─→ [Analyst] Confluence 검색         ─┤
└─→ [Analyst] Jira 이력 검색          ─┘
```

### 병렬 탐색 판단 기준 (Lead가 결정)

| 조건 | 실행 방식 |
|---|---|
| 티켓이 단순하고 runbook 매칭 예상 | 직렬 (runbook 먼저, 매칭되면 끝) |
| 티켓이 복합적이거나 매뉴얼 매칭 불확실 | 병렬 (전체 소스 동시 탐색) |
| 프로젝트성 티켓의 초기 조사 | 병렬 (다양한 관점의 자료 수집) |

## 탐색 결과 출력

Analyst가 탐색을 완료하면 아래 형식으로 Lead에게 제출한다.

```markdown
## 솔루션 탐색 결과

### 티켓 정보
- ticket_id: JIRA-1234
- 분류: 운영
- 핵심 키워드: [장애, 시스템A, 타임아웃]

### 솔루션 후보

#### 후보 1 (추천)
- **출처**: runbooks/by-system/system-a.md — 4.1절 "타임아웃 장애 대응"
- **매칭도**: 높음
- **적용 절차 요약**: [1] 로그 확인 → [2] 커넥션 풀 리셋 → [3] 서비스 헬스체크
- **과거 적용 이력**: JIRA-1100에서 적용, 성공 (2026-02-15)
- **리스크**: 커넥션 풀 리셋 시 일시적 서비스 중단 가능

#### 후보 2
- **출처**: past-solutions/2026-01-20-JIRA-0987.md
- **매칭도**: 중간
- **적용 절차 요약**: [1] 설정 변경 → [2] 서비스 재시작
- **과거 적용 이력**: JIRA-0987에서 적용, 성공 (2026-01-20)
- **리스크**: 서비스 재시작으로 다운타임 발생
- **후보 1과의 차이**: 원인이 다름 (설정 문제 vs 커넥션 풀 문제)

#### 후보 없음의 경우
- **탐색 범위**: runbooks 전체, past-solutions 전체, Confluence "OPS" 스페이스
- **결과**: 매칭되는 매뉴얼/사례 없음
- **가장 가까운 참고 자료**: [있으면 명시]
- **권장 행동**: 사용자 개입 요청 (확신도: 낮음)

### 탐색 메타 정보
- 탐색 소요 시간: 5분
- 검색한 소스: runbooks (12건), past-solutions (8건), Confluence (23페이지), Jira (15건)
```

## 탐색 품질 기준 (Guardian Gate 1)

Guardian이 Gate 1에서 검증하는 항목:

| 항목 | 기준 | 반려 사유 예시 |
|---|---|---|
| 소스 충분성 | 최소 2개 소스를 탐색했는가 | "runbook만 확인하고 과거 사례를 보지 않음" |
| 근거 명시성 | 각 후보의 출처와 매칭 근거가 구체적인가 | "매칭도: 높음"만 적고 이유가 없음 |
| 대안 제시 | 후보가 1개뿐이라면 대안이 없는 이유를 명시했는가 | 후보 1개인데 "다른 후보 없음"의 근거 부재 |
| 리스크 명시 | 각 후보의 리스크가 기술되어 있는가 | 리스크 항목 누락 |
| 최신성 | 과거 사례 참조 시 최신 사례를 우선했는가 | 1년 전 사례만 참조하고 최근 사례 무시 |

## 탐색 실패 처리

솔루션을 찾지 못한 경우의 처리 경로:

```
[Analyst] 탐색 완료, 후보 없음
  ↓
[Guardian] Gate 1: 탐색 범위 충분한가?
  │
  ├─ 불충분 → 반려: 추가 소스 탐색 지시
  │
  └─ 충분 → 통과
      ↓
  [Lead] 확신도 "낮음" 판정
      ↓
  [Lead] 사용자 개입 요청 (human-in-the-loop.md 참조)
      ↓
  [Reporter] 의사결정 로그에 "매뉴얼 갭" 기록
      ↓
  [Reporter] runbook 추가 필요 제안 → 주별 보고에 포함
```
