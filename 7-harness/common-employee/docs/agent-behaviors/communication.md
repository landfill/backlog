# 커뮤니케이션 규칙 (Communication)

## 목적

에이전트 팀이 **유관자와 어떤 시점에, 어떤 채널로, 어떤 톤으로** 소통하는지 정의한다.
커뮤니케이션은 처리의 일부다 — 유관자가 모르면 해결되지 않은 것이다 (신념 4).

## 담당 에이전트

- **커뮤니케이션 계획**: Lead (어떤 시점에 누구에게)
- **메시지 작성**: Reporter (내용)
- **메시지 발송**: Executor (Jira/Teams 웹훅), 운영자 (UI에서 수동 Outlook 메일 발송)

---

## 커뮤니케이션 시점

### 필수 알림 (모든 티켓)

| 시점 | 대상 | 채널 | 내용 |
|---|---|---|---|
| 처리 시작 | 티켓 보고자 | Jira 댓글 | "접수 확인, 분석 시작" |
| 솔루션 확정 | 티켓 보고자 | Jira 댓글 | "솔루션 확정, 처리 예정" (확신도 중간 이상) |
| 처리 완료 | 티켓 보고자 + 유관자 | Jira 댓글 | "처리 완료, 결과 요약" |
| 에스컬레이션 | 에스컬레이션 대상 | Jira 댓글 + 필요 시 운영자 수동 Outlook 메일 | "에스컬레이션 사유 + 필요한 판단" |

### 조건부 알림

| 시점 | 조건 | 대상 | 채널 |
|---|---|---|---|
| 지연 예상 | 예상 소요 시간 초과 | 티켓 보고자 | Jira 댓글 |
| 사용자 개입 필요 | 확신도 낮음 또는 분류 불명확 | 개입 대상자 | Jira 댓글 |
| 보안 이슈 | Guardian Gate 2 보안 반려 | 팀 리더 + 보안 담당 | Jira 댓글 + 필요 시 운영자 수동 Outlook 메일 |
| 프로젝트 마일스톤 | 마일스톤 완료 | 팀 리더 + 유관자 | Confluence + Jira 댓글 |

### 하지 않는 알림

- 내부 게이트 통과/반려 (에이전트 내부 프로세스, 유관자에게 알릴 필요 없음)
- 솔루션 탐색 중간 과정 (완료 후 결과만 공유)
- 매뉴얼 업데이트 제안 (주별 보고에 포함)

## 채널 선택 규칙

| 목적 | 채널 | 이유 |
|---|---|---|
| 공식 기록 | Jira 댓글 | 티켓 이력에 영구 보존 |
| 빠른 내부 알림 | Teams 웹훅 셀프 알림 | 운영자 즉시 인지 |
| 공식 보고 | Confluence 페이지 | 구조화된 보고서 |
| 긴급 + 기록 | Jira 댓글 + 운영자 수동 Outlook 메일 | 공식 기록 + 수동 메일 전달 |

### 채널 우선순위

```
1순위: Jira 댓글 (모든 처리의 기본 기록)
2순위: Confluence (정기 보고, 프로젝트 보고)
3순위: Teams 웹훅 (운영자 셀프 알림)
4순위: Outlook (운영자 수동 발송이 필요한 경우만)
```

## 톤과 형식

### 기본 톤

- 간결하고 사실 중심
- 존댓말 사용
- 기술 용어는 상황에 맞게 (유관자가 비기술인이면 풀어서)
- 감정 표현 없이 객관적으로

### 에이전트 식별

모든 메시지에 에이전트 식별자를 포함한다:
- Jira 댓글: `[🤖 팀원에이전트]`로 시작
- Teams 웹훅: `[🤖 팀원에이전트]`로 시작
- Outlook: 운영자 확인 후 발송 고지 포함

### 메시지 길이 기준

| 채널 | 최대 길이 | 상세 처리 |
|---|---|---|
| Teams 웹훅 | 500자 | 초과 시 링크로 대체 |
| Jira 댓글 | 1000자 | 초과 시 접기(collapse) 사용 |
| Outlook | 본문 자체 완결 | 링크 포함하되 메일만으로도 이해 가능 |
| Confluence | 제한 없음 | 구조화하여 가독성 확보 |

## 유관자 식별

### 유관자 범위

| 유관자 | 식별 방법 | 알림 대상 |
|---|---|---|
| 티켓 보고자 | Jira reporter 필드 | 처리 시작/완료/개입 요청 |
| 티켓 담당자 | Jira assignee 필드 (에이전트 자신 제외) | 처리 관련 |
| 팀 리더 | 고정 설정 | 에스컬레이션/보고 |
| 워처 | Jira watcher 필드 | 처리 완료 시 |
| 프로젝트 유관자 | Coordinator가 식별 | 프로젝트 보고 |

### 알림 과부하 방지

- 한 티켓에 대해 같은 사람에게 1시간 내 2회 이상 알림하지 않음
- 여러 티켓이 동시 완료된 경우 묶어서 1회 셀프 알림 가능 (Teams 웹훅)
- 일별/주별 보고가 있으므로 개별 건 알림은 간결하게

## 사용자 응답 처리

유관자가 응답해야 하는 경우의 기본 채널은 Jira 댓글이다.
Teams와 메일 응답은 시스템이 자동 수집하지 않는다.

```
[Executor] 응답 감지 (Jira 댓글)
  ↓
[Lead] 응답 내용 해석
  ├─ 명확한 지시 → 파이프라인에 반영
  ├─ 추가 질문 → Reporter가 답변 작성 → Executor 발송
  └─ 해석 불가 → 재확인 요청 (1회)
  ↓
[Reporter] 의사결정 로그에 소통 이력 기록
```
