# 티켓 분류 (Ticket Triage)

## 목적

배정된 Jira 티켓을 **운영 / 프로젝트 / 불명확**으로 분류한다.
이 분류가 이후 모든 행동의 분기점이 된다 — 잘못된 분류는 전체 파이프라인을 오염시킨다.

## 담당 에이전트

- **분류 판단**: Lead (전용 권한, 위임 불가)
- **분류 보조 정보 수집**: Analyst (Lead 요청 시)

## 분류 기준

### 1차 판단: Jira 필드 기반 (자동)

| Jira 필드 | 운영성 신호 | 프로젝트성 신호 |
|---|---|---|
| 이슈유형 | 버그, 서비스 요청, 인시던트, 작업 | 스토리, 에픽, 기능 요청 |
| 프로젝트 | 운영 프로젝트 (OPS, SUPPORT 등) | 개발 프로젝트 (DEV, FEATURE 등) |
| 우선순위 | 긴급, 높음 (즉시 대응 필요) | 보통, 낮음 (계획적 진행) |
| 레이블 | ops, incident, hotfix | project, feature, enhancement |
| 예상 소요 | 명시되지 않음 또는 1일 이내 | 1주 이상 또는 마일스톤 존재 |

### 2차 판단: 내용 분석 (AI 기반)

Jira 필드만으로 판단이 어려운 경우, 티켓 제목과 설명을 분석한다.

**운영성 키워드/패턴:**
- 장애, 오류, 에러, 긴급, 복구, 롤백, 재시작
- "~가 안 됩니다", "~가 동작하지 않습니다"
- 특정 사용자/고객의 문제 보고

**프로젝트성 키워드/패턴:**
- 신규, 개발, 기획, 설계, 마이그레이션
- "~를 추가해 주세요", "~기능 개선"
- 일정/마일스톤/릴리스 언급

### 3차 판단: 복합 상황

하나의 티켓에 운영성과 프로젝트성이 혼재하는 경우:

| 상황 | 판단 |
|---|---|
| 장애 대응 + 근본 원인 개선 | 운영성으로 시작 → 장애 해소 후 프로젝트성 후속 티켓 생성 제안 |
| 기능 요청이나 긴급도가 높음 | 내용 기반 판단 (긴급도만으로 운영성 판정하지 않음) |
| 명확히 판단 불가 | **불명확** → 사용자에게 확인 요청 |

## 분류 프로세스

```
티켓 수신
  │
  ▼
[Lead] 1차 판단: Jira 필드 분석
  │
  ├─ 필드가 일관되게 운영성 → 운영 (확신도: 높음)
  ├─ 필드가 일관되게 프로젝트성 → 프로젝트 (확신도: 높음)
  └─ 필드 신호가 혼재 또는 부족 → 2차 판단으로
  │
  ▼
[Lead] 2차 판단: 내용 분석
  │
  ├─ 내용이 명확히 분류 가능 → 운영 또는 프로젝트 (확신도: 중간~높음)
  └─ 내용도 모호 → 3차 판단으로
  │
  ▼
[Lead] 3차 판단: 복합 상황 처리
  │
  ├─ 분리 가능 → 주 분류 결정 + 후속 티켓 생성 제안
  └─ 분리 불가 → 불명확 판정 → 사용자 확인 요청
```

## 분류 결과 출력

Lead가 분류를 완료하면 아래 형식으로 기록한다. 이 기록은 의사결정 로그의 "분류 근거" 섹션에 포함된다.

```markdown
### 분류 결과

- **판단**: 운영 / 프로젝트 / 불명확
- **확신도**: 높음 / 중간 / 낮음
- **1차 근거 (필드 기반)**: 이슈유형=서비스요청, 프로젝트=OPS, 우선순위=높음
- **2차 근거 (내용 기반)**: 제목에 "장애" 키워드, 설명에 특정 에러 로그
- **Coordinator 필요**: 예 / 아니오
- **Coordinator 근거**: (프로젝트성인 경우) 예상 기간 2주, 마일스톤 3개
```

## 분류 후 분기

### 운영성 티켓

```
분류 완료 (운영)
  ↓
[Lead] 작업 DAG 생성 (단순 파이프라인)
  ↓
[Analyst] 솔루션 탐색 시작
```

### 프로젝트성 티켓

```
분류 완료 (프로젝트)
  ↓
[Lead] Coordinator 스폰 판단
  ├─ 스폰 → Coordinator가 일정 수립 후 DAG 생성
  └─ 스폰 불필요 → Lead가 직접 DAG 생성
  ↓
[Analyst] 요구사항 분석 시작
```

세부 초기화 절차와 후속 액션은 `docs/product-specs/decision-tree.md`의 프로젝트 경로를 기준으로 본다.

### 불명확 티켓

```
분류 불명확
  ↓
[Lead] 사용자에게 확인 요청
  ├─ Jira 댓글로 선택지 제시: "운영 / 프로젝트 / 기타"
  ├─ 타임아웃: 4시간
  └─ 타임아웃 시: 보류 유지 + 팀 리더 에스컬레이션
  ↓
사용자 응답 수신
  ↓
[Lead] 사용자 분류에 따라 분기
```

## 분류 실패 시

- 2차까지 분석해도 확신도 "낮음"이면 반드시 사용자에게 묻는다
- "아마 이거일 것이다"로 추측 처리하지 않는다
- 불명확 판정 자체는 실패가 아니다 — 신념 2 (모르는 것을 아는 것이 핵심 역량)

## 분류 정확도 추적

Reporter가 주별 요약에 포함하는 분류 관련 지표:

| 지표 | 설명 |
|---|---|
| 자동 분류 비율 | 1차 판단(필드 기반)으로 분류 완료된 비율 |
| 사용자 확인 비율 | 불명확 판정으로 사용자에게 물은 비율 |
| 분류 수정 비율 | 처리 중 분류가 변경된 비율 (초기 분류 오류 지표) |
| 평균 분류 소요 시간 | 티켓 수신 → 분류 완료까지의 시간 |
