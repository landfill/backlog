# 자동 처리 (Auto Resolution)

## 목적

Lead가 확정한 솔루션을 **Executor가 실제로 수행하거나 운영자 수동 액션을 위해 준비**하는 규칙을 정의한다.
자동 처리는 "확정된 계획의 실행"이지, "자의적 판단에 의한 행동"이 아니다.
세부 액션별 위험도와 승인 경로의 단일 기준은 `docs/product-specs/decision-tree.md`를 따른다.

## 담당 에이전트

- **실행**: Executor
- **실행 승인**: Lead (확신도 높음/중간일 때) 또는 사용자 (확신도 낮음일 때)
- **실행 결과 검증**: Guardian (Gate 2)
- **실행 결과 기록**: Reporter

---

## 실행 전 필수 조건

Executor는 다음 조건이 **모두** 충족되어야 실행을 시작한다:

| 조건 | 확인 방법 |
|---|---|
| Analyst의 솔루션 탐색이 완료됨 | Lead로부터 솔루션 확정 메시지 수신 |
| Guardian Gate 1을 통과함 | Guardian의 Gate 1 통과 판정 확인 |
| Lead의 실행 승인이 있음 | Lead의 위임 지시에 실행 승인 포함 |
| 확신도 낮음인 경우 사용자 승인 | 사용자 응답 확인 (human-in-the-loop 참조) |
| Reporter가 사전 로그를 작성함 | "계획" 단계 로그가 기록됨 (원칙 3: 처리보다 기록이 먼저) |

**하나라도 미충족이면 실행하지 않는다.**

---

## 실행 가능한 액션

Executor가 수행할 수 있는 액션 유형:

### Jira 액션

| 액션 | 설명 | 위험도 |
|---|---|---|
| 댓글 작성 | 조사 결과, 처리 내역, 안내사항 등 | 낮음 |
| 상태 변경 | 진행중, 해결됨, 종료 등 | 중간 |
| 필드 업데이트 | 담당자, 레이블, 우선순위, 컴포넌트 등 | 중간 |
| 티켓 생성 | 후속 티켓, 하위 작업 생성 | 중간 |
| 티켓 링크 | 관련 티켓 연결 | 낮음 |

### Confluence 액션

| 액션 | 설명 | 위험도 |
|---|---|---|
| 페이지 생성 | 보고서, 문서 신규 작성 | 중간 |
| 페이지 수정 | 기존 문서 업데이트 | 높음 |
| 댓글 작성 | 페이지에 코멘트 추가 | 낮음 |

### Teams 액션

| 액션 | 설명 | 위험도 |
|---|---|---|
| 웹훅 알림 발송 | 운영자 셀프 진행 알림 | 낮음 |

### Outlook 액션

| 액션 | 설명 | 위험도 |
|---|---|---|
| 메일 발송 준비 | 운영자 수동 SMTP 발송을 위한 초안/파라미터 준비 | 중간 |

위 표는 실행 가능한 액션 목록을 보여 주기 위한 것이고,
각 액션의 최종 위험도/승인 경로 해석은 `docs/product-specs/decision-tree.md`의 액션별 전체 매핑을 우선한다.

---

## 위험도별 실행 규칙

| 위험도 | 규칙 |
|---|---|
| **낮음** | Executor 자율 실행 가능 |
| **중간** | Executor 실행 후 즉시 Guardian Gate 2 검증 |
| **높음** | 실행 전 Lead 추가 확인 필요 (기존 문서 수정 등 비가역 작업) |

### 비가역 작업 특별 규칙

되돌리기 어려운 작업은 추가 안전장치를 적용한다:

- Confluence 기존 페이지 수정 → 수정 전 현재 내용 스냅샷 저장
- Jira 티켓 종료 → 종료 전 모든 관련 정보 로그에 기록
- 메일 발송 준비 → 운영자 수동 발송 전에 내용을 확인 가능해야 한다
- 메일 실제 발송은 운영자 액션이며, Executor는 발송 준비 evidence만 남긴다

## 실행 프로세스

```
[Lead] 실행 지시 수신
  ↓
[Executor] 전 조건 확인 (5개 항목)
  │
  ├─ 미충족 → Lead에 보고 + 대기
  │
  └─ 충족
      ↓
  [Executor] 액션 계획 확인
      │
      ├─ 위험도 높음 → Lead 추가 확인 요청
      │
      └─ 위험도 낮음/중간 → 실행 시작
          ↓
      [Executor] 액션 순차 실행
          │
          ├─ 각 액션 실행 후 결과 기록
          ├─ 에러 발생 시 → 즉시 중단 + Lead 보고
          │
          └─ 전체 완료
              ↓
          [Guardian] Gate 2: 실행 결과 + 보안 검증
              │
              ├─ 통과 → Reporter에게 전달
              │
              └─ 반려
                  ├─ 보안 이슈 → 즉시 Lead 보고 + 사용자 개입
                  └─ 품질 이슈 → Executor 재처리
```

## 실행 결과 출력

Executor가 실행을 완료하면 아래 형식으로 기록한다.

```markdown
## 실행 결과

### 실행 요약
- ticket_id: JIRA-1234
- 실행자: Executor
- 실행 시작: 2026-03-29T09:20:00
- 실행 완료: 2026-03-29T09:32:00
- 전체 결과: 성공 / 부분 성공 / 실패

### 액션 상세

| 순서 | 시각 | 대상 | 액션 | 결과 | 비고 |
|---|---|---|---|---|---|
| 1 | 09:20 | Jira | 댓글 작성 | 성공 | 조사 결과 공유 |
| 2 | 09:25 | Jira | 상태 변경 → 해결됨 | 성공 | |
| 3 | 09:30 | Teams Webhook | 셀프 알림 발송 | 성공 | 처리 완료 요약 |

### 에러 (있는 경우)
- **발생 시각**: 09:28
- **대상**: Confluence 페이지 수정
- **에러 내용**: 권한 부족 (403)
- **조치**: 실행 중단, Lead에 보고
```

## 에러 처리

### 재시도 가능한 에러

| 에러 유형 | 재시도 횟수 | 재시도 간격 |
|---|---|---|
| 네트워크 타임아웃 | 3회 | 30초, 1분, 2분 |
| API 속도 제한 (429) | 3회 | API 응답의 Retry-After 헤더 준수 |
| 일시적 서버 에러 (500, 503) | 3회 | 1분, 2분, 5분 |

### 재시도 불가능한 에러

| 에러 유형 | 즉시 행동 |
|---|---|
| 인증 실패 (401) | 실행 중단 + Lead 보고 |
| 권한 부족 (403) | 실행 중단 + Lead 보고 |
| 리소스 미존재 (404) | 실행 중단 + Lead 보고 (티켓 삭제 등) |
| 비즈니스 규칙 위반 | 실행 중단 + Lead 보고 |

### 부분 실패 처리

복수 액션 중 일부만 실패한 경우:

1. 실패 지점에서 즉시 중단 (후속 액션 실행하지 않음)
2. 이미 완료된 액션 목록을 기록
3. 실패 원인과 함께 Lead에 보고
4. Lead가 판단: 롤백 / 재시도 / 사용자 개입

## 실행 결과 검증 (Guardian Gate 2)

Guardian이 Gate 2에서 확인하는 항목:

| 항목 | 기준 |
|---|---|
| 완료성 | 계획된 모든 액션이 실행됐는가 |
| 정확성 | 실행 결과가 계획과 일치하는가 |
| 부작용 없음 | 의도하지 않은 변경이 없는가 |
| 보안 준수 | 민감 정보 노출 없는가, 권한 범위 내에서 실행됐는가 |
| 기록 완전성 | 모든 액션의 결과가 기록됐는가 |
