# 사용자 개입 프로토콜 (Human-in-the-Loop)

## 목적

에이전트 팀이 **언제, 어떻게 사람에게 묻는지**를 정의한다.
사람에게 묻는 것은 실패가 아니라 핵심 역량이다 (신념 2).
묻지 않고 틀리는 것이 묻고 늦어지는 것보다 위험하다.

## 담당 에이전트

- **개입 판단**: Lead (전용 권한, 위임 불가)
- **개입 요청 실행**: Executor (Jira 댓글 작성), 운영자 (필요 시 수동 Outlook 메일 발송)
- **개입 결과 기록**: Reporter (의사결정 로그에 기록)

## 개입 트리거

### 반드시 사람에게 묻는 경우

| 트리거 | 상황 | 근거 |
|---|---|---|
| 분류 불명확 | 티켓이 운영/프로젝트 어디에도 명확히 분류되지 않음 | 잘못된 분류는 전체 파이프라인을 오염시킴 |
| 확신도 낮음 | 솔루션 탐색 결과 확신도가 "낮음" | 근거 없는 자율 처리는 금지 |
| 매뉴얼 상충 | 동일 티켓에 적용 가능한 매뉴얼이 2개 이상이고 상충 | 에이전트가 매뉴얼 우선순위를 자의 판단하면 안 됨 |
| 에스컬레이션 대상 모호 | 에스컬레이션은 필요하나 담당자/팀이 불분명 | 잘못된 에스컬레이션은 신뢰를 훼손 |
| 보안 위반 의심 | Guardian이 Gate 2에서 보안 이슈 감지 | 보안 관련은 자율 판단하지 않음 |
| Guardian 2회 연속 반려 | 같은 작업이 Guardian에게 2회 반려됨 | 에이전트 자체로 해결 불가능한 품질 이슈 |

### 자율 처리 후 사후 확인하는 경우

| 트리거 | 상황 | 후속 |
|---|---|---|
| 확신도 중간 | 솔루션이 있으나 완전히 확실하지 않음 | 처리 후 사용자에게 결과 확인 요청 |
| 신규 패턴 | 기존 매뉴얼에 없는 처리 방식을 적용 | 처리 후 이 방식을 매뉴얼에 추가할지 확인 |
| 범위 경계 | 티켓 처리가 다른 팀의 영역에 영향을 줄 수 있음 | 처리 후 영향 확인 요청 |

### 사람에게 묻지 않는 경우

| 상황 | 이유 |
|---|---|
| 확신도 높음 + 매뉴얼 매칭 | 충분한 근거로 자율 처리 가능 |
| 과거 동일 사례 존재 + 성공 이력 | 검증된 솔루션 재사용 |
| Guardian 전체 게이트 1회 통과 | 품질이 검증된 산출물 |

## 개입 요청 프로세스

### 1단계: Jira 댓글로 확인 요청

```
[Lead] 개입 필요 판단
  ↓
[Executor] Jira 댓글 작성
```

**메시지 형식:**

```markdown
🔔 [JIRA-1234] 확인 요청

**상황**: [간결한 상황 설명]
**필요한 판단**: [구체적으로 사람이 결정해야 할 것]

선택지:
1. [선택지 A] — [선택 시 에이전트가 수행할 행동]
2. [선택지 B] — [선택 시 에이전트가 수행할 행동]
3. 직접 지시 — [사용자가 별도 지시를 내림]

⏰ 응답 기한: [시각] (미응답 시 [기본 행동])
```

**핵심 규칙:**
- 열린 질문을 하지 않는다 ("어떻게 할까요?"가 아닌 선택지를 제시)
- 각 선택지에 후속 행동을 명시한다
- 타임아웃 시 기본 행동을 미리 알린다

### 2단계: 타임아웃 처리

| 트리거 유형 | 타임아웃 | 기본 행동 |
|---|---|---|
| 분류 불명확 | 4시간 | 보류 상태 유지 + 팀 리더에게 에스컬레이션 |
| 확신도 낮음 | 8시간 | 보류 상태 유지 + Jira 댓글 리마인더 |
| 매뉴얼 상충 | 4시간 | 보수적 선택 (더 안전한 매뉴얼 적용) + 사후 확인 |
| 에스컬레이션 대상 모호 | 4시간 | 팀 리더에게 에스컬레이션 |
| 보안 위반 의심 | 즉시 | 타임아웃 없음 — 사람이 확인할 때까지 처리 중단 |
| Guardian 2회 반려 | 8시간 | 보류 + 팀 리더에게 알림 |

### 3단계: 리마인더

타임아웃의 50% 시점에 Jira 댓글 리마인더를 1회 남긴다.

```markdown
⏰ [JIRA-1234] 리마인더

이전 확인 요청에 아직 응답이 없습니다.
[미응답 시 기본 행동] 이 [시각]에 자동 적용됩니다.
```

### 4단계: 응답 수신 후

```
사용자 응답 수신
  ↓
[Lead] 응답 해석 + 작업 계획 갱신
  ↓
[Reporter] 의사결정 로그에 기록
  ├─ 개입 요청 시각
  ├─ 응답 수신 시각
  ├─ 사용자 선택 내용
  └─ 후속 행동
  ↓
파이프라인 재개
```

## 개입 채널 선택

| 긴급도 | 1차 채널 | 2차 채널 (1차 미응답 시) |
|---|---|---|
| 긴급 (보안, 장애) | Jira 댓글 | 운영자 수동 Outlook 메일 |
| 보통 (분류, 솔루션 확인) | Jira 댓글 | — (타임아웃 후 기본 행동) |
| 낮음 (사후 확인) | Jira 댓글 | — |

## 개입 요청 대상

| 상황 | 기본 대상 | 대상 불명 시 |
|---|---|---|
| 티켓 분류 확인 | 티켓 보고자 또는 담당자 | 팀 리더 |
| 솔루션 승인 | 티켓 보고자 | 팀 리더 |
| 에스컬레이션 대상 결정 | 팀 리더 | — |
| 보안 이슈 | 팀 리더 + 보안 담당 | — |
| 매뉴얼 상충 | 해당 매뉴얼 작성자 | 팀 리더 |

## 개입 빈도 관리

에이전트가 너무 자주 묻는 것도 문제다. 다음 규칙으로 개입 빈도를 관리한다.

### 동일 유형 반복 개입 감지

같은 유형의 개입 요청이 1주 내에 3회 이상 발생하면:
1. Reporter가 패턴을 기록한다
2. Lead가 주별 보고에 포함한다
3. 매뉴얼 추가 또는 분류 규칙 갱신을 제안한다

### 개입률 목표

| 지표 | 목표 | 비고 |
|---|---|---|
| 전체 개입률 | < 20% | 10건 중 8건은 자율 처리 |
| 분류 개입률 | < 10% | 대부분 자동 분류 가능해야 함 |
| 동일 유형 반복 | 0건 | 같은 이유로 반복해서 묻지 않음 |

목표 미달 시 원인을 분석하고 매뉴얼/규칙을 보강한다.
