# 의사결정 로그 (Decision Logging)

## 목적

모든 자율 처리에는 "왜 그렇게 판단했는가"를 기록한다.
이 로그는 사후 변명이 아니라 **사전 계획 + 사후 결과**의 쌍으로 구성된다.
보고서의 원천 데이터이자, 에이전트 판단력 개선의 피드백 루프다.

## 담당 에이전트

- **작성**: Reporter
- **입력 제공**: Lead (분류/계획), Analyst (분석 결과), Executor (실행 결과), Guardian (게이트 판정)
- **읽기**: 전체 에이전트

## 로그 파일 규칙

### 저장 위치

```
docs/generated/decision-logs/
├── 2026/
│   └── 03/
│       ├── 2026-03-29-JIRA-1234.md
│       └── 2026-03-29-JIRA-1235.md
└── _summary/
    ├── weekly-2026-W13.md
    └── monthly-2026-03.md
```

### 파일명 규칙

- 형식: `YYYY-MM-DD-TICKET-ID.md`
- 날짜는 **처리 시작일** 기준
- 동일 티켓의 추가 처리는 같은 파일에 누적 (섹션 추가)

### 하나의 티켓에 복수 처리가 발생한 경우

파일 내에 `## 처리 #N` 섹션을 추가한다. 파일을 분리하지 않는다.

---

## 로그 스키마

매 처리 건마다 아래 구조를 따른다.

```markdown
# JIRA-1234: [티켓 제목]

## 메타 정보

| 항목 | 값 |
|---|---|
| ticket_id | JIRA-1234 |
| ticket_url | https://jira.example.com/browse/JIRA-1234 |
| 분류 | 운영 / 프로젝트 |
| 분류 방식 | 자동판별 / 사용자지정 |
| 처리 시작 | 2026-03-29T09:00:00 |
| 처리 완료 | 2026-03-29T09:45:00 |
| 최종 상태 | 해결 / 에스컬레이션 / 보류 / 진행중 |

## 참여 에이전트

| 에이전트 | 역할 | 작업 내용 |
|---|---|---|
| Lead | 분류 + 계획 | 운영성 판정, DAG 생성 |
| Analyst | 조사 | runbook 매칭, 과거 사례 3건 검색 |
| Guardian | 검증 | Gate 1 통과, Gate 2 통과, Gate 3 통과 |
| Executor | 실행 | Jira 상태 변경, 댓글 작성 |
| Reporter | 기록 | 본 로그 작성, 유관자 알림 |

## 판단 근거

### 분류 근거 (Lead)

- **판단**: 운영성 티켓
- **근거**: 이슈유형 "서비스 요청", 프로젝트 "OPS", 제목에 "장애" 키워드
- **확신도**: 높음

### 솔루션 탐색 결과 (Analyst)

#### 참조한 매뉴얼
- `runbooks/by-ticket-type/service-request.md` — 3.2절 "장애 대응 절차" (매칭도: 높음)

#### 참조한 과거 사례
- `past-solutions/2026-02-15-JIRA-1100.md` — 유사도: 85%, 동일 시스템 장애
- `past-solutions/2026-01-20-JIRA-0987.md` — 유사도: 60%, 유사 증상

#### 외부 검색 결과
- Confluence: "시스템A 장애 대응 가이드" (wiki-page-id: 12345)
- Jira: 유사 티켓 2건 (JIRA-1100, JIRA-0987)

### 솔루션 선택 (Lead)

- **선택한 솔루션**: runbook 3.2절 절차 적용
- **선택 근거**: 매뉴얼 매칭도 높음 + 과거 사례 JIRA-1100과 동일 패턴
- **확신도**: 높음
- **대안**: 과거 사례 JIRA-0987의 방법 (채택하지 않은 이유: 증상은 유사하나 원인이 다름)

## 게이트 판정 (Guardian)

### Gate 1: 분석 품질
- **결과**: 통과
- **판정 시각**: 2026-03-29T09:15:00
- **비고**: 매뉴얼 매칭 근거 충분, 과거 사례 2건 이상 확보

### Gate 2: 실행 결과 + 보안
- **결과**: 통과
- **판정 시각**: 2026-03-29T09:35:00
- **보안 체크**: 위반 사항 없음
- **비고**: Jira 상태 변경 정상, 의도치 않은 필드 변경 없음

### Gate 3: 보고 품질
- **결과**: 통과
- **판정 시각**: 2026-03-29T09:42:00
- **비고**: 로그 스키마 준수, 유관자 알림 발송 확인

## 실행 내역 (Executor)

| 시각 | 대상 시스템 | 액션 | 상세 |
|---|---|---|---|
| 09:20 | Jira | 댓글 작성 | 조사 결과 및 처리 계획 공유 |
| 09:30 | Jira | 상태 변경 | "진행중" → "해결됨" |
| 09:31 | Teams | 알림 발송 | 유관자에게 처리 완료 알림 |

## 결과

- **최종 상태**: 해결
- **처리 소요 시간**: 45분
- **사용자 개입 여부**: 없음

## 학습 포인트

- **새로운 패턴 발견**: 없음 (기존 매뉴얼로 처리 완료)
- **매뉴얼 업데이트 필요**: 없음
- **특이 사항**: 없음
```

## 확신도 기준

Lead가 솔루션을 선택할 때 부여하는 확신도 기준:

| 확신도 | 조건 | 후속 행동 |
|---|---|---|
| **높음** | 매뉴얼이 정확히 매칭 + 과거 사례가 1건 이상 성공 | 자율 처리, 사후 보고 |
| **중간** | 매뉴얼이 부분 매칭 또는 과거 사례가 유사하나 동일하지 않음 | 자율 처리 + 사후 확인 요청 |
| **낮음** | 매뉴얼 없음 + 과거 사례 없거나 유사도 낮음 | 솔루션 제안 + 사람 승인 대기 |

## 요약 보고서 자동 생성 규칙

### 주별 요약 (`_summary/weekly-YYYY-WNN.md`)

Reporter가 매주 생성. 포함 항목:

- 처리 건수 (총/자율처리/사람개입)
- 확신도별 분포 (높음/중간/낮음)
- 평균 처리 시간
- Guardian 반려 건수 및 사유 요약
- 새로운 패턴 발견 목록
- 매뉴얼 업데이트 제안 목록

### 월별 요약 (`_summary/monthly-YYYY-MM.md`)

주별 요약을 집계. 추가 항목:

- 자율 처리 비율 추이
- 가장 많이 참조된 매뉴얼 TOP 5
- 가장 많이 재사용된 과거 사례 TOP 5
- 반복되는 Guardian 반려 패턴
