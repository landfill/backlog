# 티켓 → 액션 의사결정 트리 (Decision Tree)

## 목적

이 문서는 `common-employee` 앱이 티켓을 어떤 처리 경로와 어떤 액션 세트로 보낼지 결정하는 단일 기준이다.
`docs/product-specs/ticket-lifecycle.md`가 전체 단계 순서를 정의한다면, 이 문서는 단계 안에서 어떤 분기를 선택할지를 고정한다.

## 기준 연결

- 처리 순서: `docs/product-specs/ticket-lifecycle.md`
- 분류 규칙: `docs/agent-behaviors/ticket-triage.md`
- 탐색 규칙: `docs/agent-behaviors/solution-lookup.md`
- 실행 규칙: `docs/agent-behaviors/auto-resolution.md`
- 사람 개입: `docs/agent-behaviors/human-in-the-loop.md`
- 품질 판정: `docs/product-specs/quality-criteria.md`
- 데이터 흐름과 저장/공유 경계: `docs/design-docs/data-flow.md`
- 구현 전 고정 범위: `docs/design-docs/design-freeze-matrix.md`

## 입력

- 티켓 분류: 운영 / 프로젝트 / 불명확
- 확신도: 높음 / 중간 / 낮음
- 솔루션 탐색 결과: 매칭 있음 / 부분 매칭 / 후보 없음
- 액션 위험도: 낮음 / 중간 / 높음
- Guardian gate 결과: `APPROVED` / `CHANGES_REQUESTED` / `BLOCKED` / `SKIPPED`

## 전체 트리

```text
티켓 수신
  ↓
[1] 분류 결정
  ├─ 운영
  │   ↓
  │ [2] 탐색 방식 결정
  │   ├─ runbook 정확 매칭 예상 → 직렬 탐색
  │   └─ 매칭 불확실 / 복합 징후 → 병렬 탐색
  │   ↓
  │ [3] 솔루션 상태 결정
  │   ├─ 후보 있음 + Gate 1 통과
  │   │   ↓
  │   │ [4] 실행 승인 경로
  │   │   ├─ 확신도 높음 → Lead 승인 후 실행
  │   │   ├─ 확신도 중간 → Lead 승인 후 실행 + 사후 확인 예약
  │   │   └─ 확신도 낮음 → 사람 승인 대기
  │   │   ↓
  │   │ [5] 액션 위험도 경로
  │   │   ├─ 낮음 → Executor 실행
  │   │   ├─ 중간 → Executor 실행 후 Gate 2
  │   │   └─ 높음/비가역 → Lead 추가 확인 후 실행
  │   │   ↓
  │   │ [6] 결과 처리
  │   │   ├─ Gate 2/3 통과 → 해결 또는 후속 액션 연결
  │   │   └─ 반려/차단 → 같은 단계 재작업 또는 사람 개입
  │   │
  │   └─ 후보 없음 → 사람 개입 + 매뉴얼 갭 기록
  │
  ├─ 프로젝트
  │   ↓
  │ [2] Coordinator 필요 여부
  │   ├─ 필요 → Coordinator 스폰 + exec-plan 생성 + 마일스톤/하위 작업/보고 주기 초기화
  │   └─ 불필요 → Lead 직접 조율 + 축소된 프로젝트 관리 적용
  │   ↓
  │ [3] 마일스톤 단위로 운영 트리 반복
  │
  └─ 불명확
      ↓
    사람에게 분류 확인 요청
      ├─ 응답 수신 → 운영 / 프로젝트 분기
      └─ 타임아웃 → `docs/agent-behaviors/human-in-the-loop.md`의 타임아웃 규칙에 따라 Lead 최선 추정 + 사후 확인 예약
```

## 1. 분류 결정

### 운영 경로

아래 중 하나라도 강하게 성립하면 운영 경로를 우선한다.

- 장애, 오류, 복구, 롤백, 즉시 대응 중심의 티켓
- 1일 이내 단일 처리 사이클로 닫힐 수 있는 작업
- 기존 runbook 또는 past-solution으로 해결 가능성이 높은 작업

### 프로젝트 경로

아래 중 하나라도 성립하면 프로젝트 경로를 우선한다.

- 신규 기능, 설계, 마이그레이션, 개선 작업
- 1주 이상 또는 복수 마일스톤이 필요한 작업
- 여러 하위 작업과 보고 주기가 필요한 작업

### 불명확 경로

아래 조건이면 사람에게 먼저 묻는다.

- 운영성과 프로젝트성이 동시에 강하다
- 티켓 설명만으로 영향 범위를 판단할 수 없다
- 잘못 분류하면 실행 위험이 커진다

## 2. 탐색 방식 결정

### 직렬 탐색 조건

- runbook 정확 매칭 가능성이 높다
- 티켓 구조가 단순하다
- 빠른 복구가 더 중요하다

### 병렬 탐색 조건

- 매뉴얼 매칭이 불확실하다
- 복합 원인 가능성이 높다
- 프로젝트 초기 조사이거나 다양한 근거가 필요하다

## 2-A. 프로젝트 티켓 초기화 경로

### Coordinator를 스폰하는 경우

아래 순서를 같은 초기화 묶음으로 본다.

1. Coordinator 스폰
2. `docs/exec-plans/active/YYYY-MM-DD-<slug>-exec-plan.md` 생성
3. 마일스톤 정의
4. 마일스톤별 Jira 하위 작업 생성
5. 프로젝트 보고 주기 설정
6. tracker 대표 상태 시작
7. 마일스톤별 task brief / ongoing plan 시작

적용 조건:
- 예상 처리 기간 1주 이상
- 하위 작업 3개 이상
- 정기 보고 2회 이상 필요
- Lead 직접 조율보다 조율 비용이 크다

### Coordinator를 스폰하지 않는 경우

아래 순서를 축소된 프로젝트 관리로 본다.

1. Lead가 직접 프로젝트 DAG 생성
2. 필요 최소한의 마일스톤만 task brief에 고정
3. Jira 하위 작업은 선택적으로 생성
4. 별도 프로젝트 보고 없이 일반 일별/주별 보고에 포함
5. tracker와 ongoing plan으로 상태를 관리

적용 조건:
- 1일 이내 또는 매우 짧은 기간 안에 끝날 수 있다
- 마일스톤 2~3개 이내로 충분하다
- 선형 파이프라인으로 관리 가능하다

구현 규칙:
- 프로젝트 티켓은 Coordinator 유무와 관계없이 `운영 상태 산출물`을 반드시 남긴다
- Coordinator를 스폰한 경우 `exec-plan`이 프로젝트 초기화의 공식 계획 문서다
- Coordinator를 스폰하지 않은 경우 별도 `exec-plan`은 선택 사항이고 task brief가 최소 계획 문서가 된다

## 3. 솔루션 상태 결정

### 후보 있음

아래를 모두 만족하면 실행 준비 가능 상태로 본다.

- 우선 근거 1개 이상
- 보조 근거 1개 이상
- 리스크 또는 확인 포인트 명시
- Gate 1 `APPROVED`

### 후보 없음

아래 행동을 같이 묶어 실행한다.

- Lead가 확신도 `낮음`으로 판정
- 사람 개입 요청
- Reporter가 decision log에 매뉴얼 갭 기록
- feedback-loop에서 runbook 후보 검토 대상으로 올림

## 4. 실행 승인 경로

### 확신도 높음

- Lead가 실행 승인한다
- Executor는 승인 범위 안에서 액션을 수행한다
- 사후 보고만 요구한다

### 확신도 중간

- Lead가 실행 승인한다
- Executor가 액션을 수행한다
- Lead 또는 Reporter가 사후 확인 예약을 남긴다

### 확신도 낮음

- 사람 승인 전까지 실행하지 않는다
- 사람의 선택지가 없으면 실행 지시로 보지 않는다

## 5. 액션 위험도 경로

세부 액션별 위험도와 승인 경로의 단일 기준은 이 섹션이다.
`docs/agent-behaviors/auto-resolution.md`는 아래 기준을 실행 규칙으로 설명하는 문서다.

### 낮음

- 댓글 작성
- 티켓 링크
- 일반 알림 메시지

처리 규칙:
- Executor 자율 실행 가능
- 결과는 ongoing plan과 실행 기록에 남긴다

### 중간

- Jira 상태 변경
- 필드 업데이트
- 하위 작업 생성
- 일반 메일 발송

처리 규칙:
- 승인 흔적 확인 후 실행
- 실행 직후 Gate 2 검증

### 높음 또는 비가역

- Confluence 기존 페이지 수정
- 티켓 종료
- 되돌리기 어려운 공지/메일 발송

처리 규칙:
- Lead 추가 확인이 있어야 한다
- 사전 스냅샷 또는 현재 상태 보존 흔적이 있어야 한다

### 액션별 전체 매핑

| 시스템 | 액션 | 위험도 | 승인 경로 | 추가 조건 |
|---|---|---|---|---|
| Jira | 댓글 작성 | 낮음 | 확정된 계획 범위 안에서 Executor 실행 | 결과 기록 필요 |
| Jira | 상태 변경 | 중간 | Lead 승인 또는 낮음 확신도일 때 사용자 승인 후 실행 | Gate 2 검증 |
| Jira | 필드 업데이트 | 중간 | Lead 승인 또는 낮음 확신도일 때 사용자 승인 후 실행 | Gate 2 검증 |
| Jira | 티켓 생성 | 중간 | Lead 승인 후 실행 | 후속 티켓/하위 작업 목적 기록 |
| Jira | 티켓 링크 | 낮음 | 확정된 계획 범위 안에서 Executor 실행 | 링크 대상 기록 |
| Confluence | 페이지 생성 | 중간 | Lead 승인 후 실행 | Gate 2 검증 |
| Confluence | 페이지 수정 | 높음 | Lead 추가 확인 후 실행 | 수정 전 현재 상태 보존 |
| Confluence | 댓글 작성 | 낮음 | 확정된 계획 범위 안에서 Executor 실행 | 결과 기록 필요 |
| Teams | 메시지 발송 | 낮음 | 확정된 계획 범위 안에서 Executor 실행 | 발송 결과 기록 |
| Teams | 채널 메시지 | 중간 | Lead 승인 후 실행 | 공유 채널 기록 |
| Outlook | 메일 발송 | 중간 | Lead 승인 후 실행 | 일반 메일은 발송 전 내용 확인, 긴급 에스컬레이션은 예외 가능 |

해석 규칙:
- 이 표에 없는 새 액션은 구현 단계에서 임의 추가하지 않고 app-level 설계 문서에 먼저 반영한다
- 액션이 둘 이상의 시스템에 걸치면 가장 높은 위험도를 따른다
- 액션 목적이 같아도 외부 공개 범위가 커지면 더 엄격한 승인 경로를 따른다

## 6. 결과 처리

### 정상 완료

아래를 모두 만족하면 완료 후보로 올린다.

- Gate 2 `APPROVED`
- Gate 3 `APPROVED`
- tracker / ongoing plan / review report 최신화
- unresolved item은 next action 또는 backlog에 연결

### 재작업

아래면 같은 단계로 되돌린다.

- `CHANGES_REQUESTED`
- 충분한 근거 없이 보완 가능하다

### 차단

아래면 자동 진행을 멈춘다.

- 보안 이슈
- 인증/권한 문제
- 외부 쓰기 결과 불명확
- 같은 root cause 반복

## 7. 후속 티켓 / 후속 문서 결정

### 후속 프로젝트 티켓 생성 제안

아래면 운영 티켓을 닫더라도 프로젝트 후속을 제안한다.

- 장애 대응은 끝났지만 근본 원인 개선이 남아 있다
- 반복되는 수동 작업을 구조 개선으로 바꿔야 한다
- 정책/프로세스 변경이 필요하다

### 문서 보강 연결

아래면 feedback-loop에 연결한다.

- 매뉴얼 갭이 드러난다
- 기존 용어가 모호하다
- 같은 에스컬레이션 사유가 반복된다

## 구현 경계

이 문서는 구현자가 따라야 할 분기 기준을 고정한다.
이 문서 자체는 구현 전에 잠가야 하는 Blocker 기준이다.

구현 단계에서 바꿔도 되는 것:
- 내부 클래스/모듈 이름
- 로그 저장 세부 포맷
- 검색 질의 세부 최적화
- UI/콘솔 표현 방식

구현 단계에서 바꾸면 안 되는 것:
- 운영 / 프로젝트 / 불명확 분기 구조
- 확신도별 승인 경로
- 위험도별 실행 경로
- 사람 개입이 필요한 조건
- feedback-loop로 연결되는 조건

세부 구분은 `docs/design-docs/design-freeze-matrix.md`를 단일 기준으로 따른다.
