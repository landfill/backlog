# 처리 품질 기준 (Quality Criteria)

## 목적

이 문서는 `common-employee` 앱이 티켓이나 프로젝트 마일스톤을 처리할 때
무엇을 "통과"와 "반려"로 볼지 정의한다.

처리 **순서**는 `docs/product-specs/ticket-lifecycle.md`를 기준으로 하고,
이 문서는 각 단계의 **판정 기준**만 담당한다.

## 평가 단위

- 운영 티켓 1건
- 프로젝트 티켓의 마일스톤 1건

## 단일 기준 문서

- 처리 순서: `docs/product-specs/ticket-lifecycle.md`
- 역할 경계: `docs/design-docs/agent-roles.md`
- 데이터 흐름과 상태 산출물별 허용 정보: `docs/design-docs/data-flow.md`
- 보안 판정: `docs/SECURITY.md`
- 신뢰성 판정: `docs/RELIABILITY.md`
- 상태 산출물 형식: `docs/status/README.md` 및 `harness/core/templates/`

## 품질 판정 레벨

| 레벨 | 의미 | 기본 판정 |
|---|---|---|
| Blocker | 안전성, 데이터 보호, 범위 통제가 깨짐 | `BLOCKED` |
| Major | 다음 단계가 이어받기 어려울 정도로 근거나 결과가 부족함 | `CHANGES_REQUESTED` |
| Minor | 품질은 낮지만 현재 단계 완료 여부를 뒤집지는 않음 | `APPROVED` + open risk 기록 |

이 문서의 `Blocker`는 gate 판정용 이슈 심각도다.
구현 전 설계 잠금 레벨은 `docs/design-docs/design-freeze-matrix.md`의 `Blocker`를 따른다.

## 공통 통과 조건

모든 단계는 아래 조건을 공통으로 만족해야 한다.

- 현재 단계의 목적과 산출물이 task brief 범위 안에 있다.
- evidence가 실제 확인 가능한 근거를 가리킨다.
- 상태 산출물 이름과 역할 이름이 기존 문서와 일치한다.
- 다음 owner와 다음 step이 한 가지 행동으로 적혀 있다.
- 남은 위험이 있으면 숨기지 않고 review report 또는 ongoing plan에 남긴다.

## Gate 1: 분석 품질 기준

Analyst의 탐색 결과를 Guardian이 검증한다.

### 필수 evidence

- 분류 결과와 확신도
- 최소 1개의 우선 기준 문서 또는 운영 근거
- 최소 1개의 보조 근거
- 선택하지 않은 대안 또는 보류 이유
- 실행 전 위험 또는 확인 포인트

### 통과 기준

| 항목 | 통과 기준 |
|---|---|
| 문제 이해 | 티켓 문제를 한 문장으로 다시 설명할 수 있다 |
| 근거 품질 | 참조한 문서/사례/외부 시스템 결과가 명시돼 있다 |
| 범위 통제 | 지금 단계에서 할 것과 하지 않을 것이 드러난다 |
| 실행 준비 | 다음 단계가 바로 실행할 수 있게 precondition이 정리돼 있다 |
| 불확실성 처리 | 확신도 낮음이면 사람 개입 조건이 함께 적혀 있다 |

### 반려 조건

- 참조 근거가 없거나 "경험상" 수준으로만 적혀 있다
- 솔루션 후보가 하나뿐인데 대안 부재 이유가 없다
- 실행 리스크나 확인 포인트가 비어 있다
- 확신도 낮음인데도 사람 승인 조건이 없다

## Gate 2: 실행 결과 + 보안 기준

Executor의 실행 결과를 Guardian이 검증한다.

### 필수 evidence

- Gate 1 통과 기록
- Lead 또는 Coordinator 승인 흔적
- 실행 대상 시스템, 액션, 결과
- 보안 확인 결과
- cleanup 상태

### 통과 기준

| 항목 | 통과 기준 |
|---|---|
| 승인 선행 | Gate 1 통과와 실행 승인 흔적이 있다 |
| 실행 완결성 | 무엇을 어디에 적용했는지가 남아 있다 |
| 범위 일치 | 실제 실행이 승인된 범위를 넘지 않는다 |
| 보안 준수 | 민감 정보 노출, 과권한 사용, 의도치 않은 변경이 없다 |
| 복구 가능성 | 실패 또는 중단 시 다음 단계가 상태를 복원할 수 있다 |

### 반려 조건

- 외부 시스템 쓰기 작업이 기록 없이 수행됐다
- 실행 결과가 모호해 실제 반영 여부를 설명할 수 없다
- 민감 정보가 로그, 문서, 메시지에 그대로 남았다
- 승인된 범위 밖 변경이 발생했다
- cleanup 상태가 없거나 다음 단계 진행이 안전하지 않다

## Gate 3: 보고 품질 기준

Reporter의 로그와 대외 공유 결과를 Guardian이 검증한다.

### 필수 evidence

- 의사결정 로그 링크 또는 파일 경로
- 유관자 공유 채널과 공유 결과
- tracker / ongoing plan 최신화 흔적
- unresolved issue 또는 next action

### 통과 기준

| 항목 | 통과 기준 |
|---|---|
| 스키마 준수 | 로그와 보고서가 정해진 형식을 따른다 |
| 근거 보존 | 누가 무엇을 왜 판단했는지가 추적 가능하다 |
| 수신자 적합성 | 외부 공유 메시지가 짧고 이해 가능하다 |
| 상태 정합성 | tracker, ongoing plan, review report의 상태가 서로 맞는다 |
| 미완료 처리 | 남은 항목은 완료로 숨기지 않고 open risk나 next action으로 남긴다 |

### 반려 조건

- verdict만 있고 evidence가 없다
- 외부 공유 내용이 실제 실행 결과와 다르다
- 상태 산출물이 최신이 아니라 다음 단계가 혼동된다
- 미완료 항목이 있는데 완료로 보고한다

## 완료 기준

운영 티켓 또는 프로젝트 마일스톤은 아래를 만족할 때만 완료 후보가 된다.

- 필요한 gate가 모두 `APPROVED`다
- `tracker.md`의 verdict, evidence, next action이 최신이다
- ongoing plan에 마지막 evidence와 남은 이슈가 남아 있다
- unresolved item은 backlog 또는 follow-up action으로 연결됐다
- 마지막 owner가 cleanup 상태를 기록했다

## 측정 지표

착수 후 품질 운영은 아래 지표를 기본으로 본다.

| 지표 | 목표 |
|---|---|
| 의사결정 로그 생성률 | 100% |
| review report 생성률 | 100% |
| evidence 없는 verdict 비율 | 0% |
| 추적 불가능한 외부 쓰기 비율 | 0% |
| 민감 정보 노출 건수 | 0건 |
| 단계 변경 후 상태 문서 미갱신 건수 | 0건 |

## 판정 기록 규칙

- `APPROVED`는 다음 단계가 바로 이어받을 수 있을 때만 쓴다.
- `CHANGES_REQUESTED`는 같은 단계 재작업이 가능할 때 쓴다.
- `BLOCKED`는 외부 승인, 보안 이슈, 인증 이슈, 문서 공백 때문에 멈출 때 쓴다.
- `SKIPPED`는 현재 task brief 범위에 적용되지 않을 때만 쓴다.

## 착수 전 차단 조건

아래 중 하나라도 남아 있으면 프로젝트 착수 준비가 완료되지 않은 것으로 본다.

- Gate 기준 문서가 없어 검수자마다 다른 판단을 한다
- 보안/신뢰성 기준이 app-level로 내려오지 않았다
- 상태 산출물 없이 작업을 넘기는 예외를 허용한다
- backlog 문서와 실제 탐색 경로가 어긋난다
