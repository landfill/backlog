# Rollback Loop

## 1. Purpose
이 문서는 같은 문제로 작업이 반복 실패할 때의 공통 복구 흐름을 정의한다.
목표는 실패를 숨기지 않고, 더 작은 단위로 다시 검증하는 것이다.

## 2. Trigger
- 아래 중 하나라도 만족하면 이 흐름을 사용한다.
1. 같은 `root_cause`가 3회 반복되었다.
2. 현재 작업의 시도 횟수가 5에 도달했는데도 문제가 남아 있다.
- 시도 횟수는 `CHANGES_REQUESTED` 또는 `BLOCKED` 판정이 나올 때마다 기록한다.
- rollback loop에 들어가면 현재 시도를 종료하고 이 흐름을 우선 적용한다.

## 3. Default Steps
1. 실패 원인을 짧게 적는다.
2. task brief, tracker, ongoing plan, 마지막 review report를 함께 본다.
3. 문서, 구현, 테스트 중 어디가 어긋났는지 구분한다.
4. 현재 작업의 시도 횟수와 마지막 판정 기록을 확인한다.
5. `checkpoint_ref` 중 되돌릴 지점을 고른다.
6. 범위나 요구사항을 더 작은 단위로 줄인다.
7. 필요한 문서를 먼저 고친다.
8. 더 작은 범위로 다시 한 사이클을 실행한다.

## 4. Rule
- 먼저 자율적으로 원인을 좁히고 수정한다.
- 같은 시도를 반복하지 않는다.
- 실패를 임시 우회로 덮지 않는다.
- 범위를 줄일 때도 검증 가능성은 유지한다.
- 보류한 항목은 사라지지 않게 남긴다.
- rollback 후 다음 단계는 한 가지 행동으로 다시 적는다.

## 5. Output
- 시도 횟수
- 실패 원인
- 사용한 `checkpoint_ref`
- 줄인 범위
- 다시 시도할 기준
- 보류된 항목
