# Repository Reliability

## 1. Purpose
이 문서는 저장소 전체에 공통으로 적용되는 검증 최소 기준을 정의한다.
모든 앱은 자기 도메인에 맞는 검증 규칙을 추가할 수 있다.

## 2. Core Rules
- 완료 기준은 검증 가능해야 한다.
- 문서, 구현, 검증 결과는 같은 동작을 가리켜야 한다.
- 검증이 없는 기능은 완료로 보지 않는다.

## 3. Default Evidence
- 목적과 범위가 문서에 적혀 있어야 한다.
- 핵심 동작은 테스트로 확인할 수 있어야 한다.
- 자동 검증이 어려운 항목은 리뷰 기준이 있어야 한다.
- 실패했을 때 무엇이 기대값과 달랐는지 드러나야 한다.

## 4. App-Level Rule
- repository-level reliability는 모든 앱의 기본선이다.
- app-level reliability는 도메인별 엣지 케이스와 기대 결과를 정의한다.
- 충돌 시 더 엄격하고 더 검증 가능한 기준을 따른다.

## 5. Failure Handling
- 검증 실패는 무시하지 않는다.
- 실패 원인은 문서, 구현, 테스트 중 어디가 어긋났는지로 설명해야 한다.
- 같은 실패가 반복되면 요구사항이나 구조를 줄여 다시 검증한다.
